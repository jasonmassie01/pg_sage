## 1. Goals

1. Reduce Mean Time To Resolution (MTTR) by telling operators **why** things are broken, not just **what** is broken.
2. Prevent cascading failures from lock chains and runaway queries before they exhaust connections.
3. Give developers (not just DBAs) the ability to understand query performance through plain-English explanations.
4. Forecast storage exhaustion before it causes unplanned outages.

### Non-Goals

- No new UI pages (dashboard changes are out of scope; API endpoints only).
- No changes to the trust ramp model or executor safety tiers.
- No MCP server integration.
- No replication failover automation.
- No query rewriting (v0.10 scope).

## 3. Feature 1: Root Cause Analysis Engine

### 3.1 Problem

Current alerts fire on symptoms: "connections high", "cache hit ratio low", "replication lag increasing". Operators see 47 alerts and none say why. Investigation takes 40+ minutes for what turns out to be a 30-second fix.

### 3.2 Design

A two-tier correlation engine that runs after every analyzer cycle completes.

**Tier 1 — Deterministic Decision Tree** (covers ~70% of incidents):

Each signal fires when a concrete threshold is breached. Thresholds derive from the existing `AnalyzerConfig` where possible, and from new `RCAConfig` fields otherwise.

#### Signal Definitions and Thresholds

| Signal ID | Fires When | Source Fields | Threshold |
|-----------|-----------|---------------|-----------|
| `connections_high` | `TotalBackends / MaxConnections > 0.80` | `System.TotalBackends`, `System.MaxConnections` | 80% of max_connections (configurable: `rca.connection_saturation_pct`, default 80) |
| `idle_in_tx_elevated` | `IdleInTransaction > 5` AND any idle-in-tx session age > `analyzer.idle_in_transaction_timeout_minutes` | `System.IdleInTransaction`, `ConfigData.ConnectionStates` | Reuses existing `AnalyzerConfig.IdleInTxTimeoutMinutes` |
| `cache_hit_ratio_drop` | Current `CacheHitRatio` < previous by > 5% absolute OR below `analyzer.cache_hit_ratio_warning` | `System.CacheHitRatio`, previous `System.CacheHitRatio` | Reuses existing `AnalyzerConfig.CacheHitRatioWarning` + 5% delta |
| `replication_lag_increasing` | Any replica `ReplayLag` parses to > 30s AND increased from previous snapshot | `Replication.Replicas[].ReplayLag` | 30s default (configurable: `rca.replication_lag_threshold_seconds`, default 30) |
| `vacuum_blocked` | Tables with `NDeadTup / (NLiveTup + NDeadTup) > analyzer.table_bloat_dead_tuple_pct / 100` AND `LastAutovacuum` > 2 hours ago | `Tables[].NDeadTup`, `Tables[].NLiveTup`, `Tables[].LastAutovacuum` | Reuses `AnalyzerConfig.TableBloatDeadTuplePct` |
| `query_regression` | Any query `MeanExecTime` increased > `analyzer.regression_threshold_pct`% vs previous snapshot | `Queries[]` delta | Reuses `AnalyzerConfig.RegressionThresholdPct` |
| `disk_pressure` | `DBSizeBytes / disk_capacity > safety.disk_pressure_threshold_pct / 100` | `System.DBSizeBytes` | Reuses `SafetyConfig.DiskPressureThresholdPct` |
| `lock_contention` | Lock chain feature (Feature 2) detected chains with `total_blocked >= lock_chain.min_blocked_threshold` | Lock chain findings | Consumes Feature 2 output |
| `wal_growth_spike` | Sum of `WALBytes` across all queries increased > 2x from previous snapshot | `Queries[].WALBytes` delta | 2x multiplier (configurable: `rca.wal_spike_multiplier`, default 2.0) |

#### Decision Trees

```
Signal: connections_high
  -> Check: idle_in_tx_elevated also firing?
    -> YES: Find PID with longest idle-in-tx duration from ConfigData.ConnectionStates
      -> Check: PID appears in Locks[] with Granted=true?
        -> YES: ROOT CAUSE: "Idle-in-transaction PID {pid} holding {lock_mode} on
                 {table} for {duration}. Blocking {n} sessions. Dead tuples
                 accumulating: {dead_tuples} on {affected_tables}."
        -> NO:  ROOT CAUSE: "Idle-in-transaction PID {pid} open for {duration}.
                 Consuming {n} of {max} connection slots but not holding locks."
    -> NO: Check ConfigData.ConnectionChurn > 2x previous snapshot?
      -> YES: ROOT CAUSE: "Connection storm: {churn_delta} new connections in
               last cycle (was {prev_churn}). Application connection pool may be
               misconfigured or restarting."
      -> NO: Check single query dominating ActiveBackends?
              (query with Calls delta > 50% of TotalBackends)
        -> YES: ROOT CAUSE: "Query {queryid} consuming {n} active backends.
                 Mean exec time: {mean}ms (was {prev_mean}ms)."
        -> NO:  ROOT CAUSE: "Gradual connection growth. TotalBackends={n},
                 MaxConnections={max}. Review application pool sizing."

Signal: cache_hit_ratio_drop
  -> Check: Any query's SharedBlksRead delta > 10x its previous value?
    -> YES: Find query with largest SharedBlksRead increase
      -> ROOT CAUSE: "Query {queryid} reading {blocks_per_call} blocks/call
         (was {prev_blocks}). {diagnosis}."
         Where diagnosis = "New query pattern" if query is new to pg_stat_statements,
         or "Possible plan regression" if query existed previously.
    -> NO: Check System.DBSizeBytes > shared_buffers * 4?
           (shared_buffers from ConfigData.PGSettings)
      -> YES: ROOT CAUSE: "Working set (~{working_set_mb}MB estimated from
               SharedBlksRead across all queries) likely exceeds shared_buffers
               ({shared_buffers_mb}MB). Consider increasing shared_buffers."
      -> NO:  ROOT CAUSE: "Cache hit ratio dropped from {prev}% to {curr}%.
               No single query or working set issue identified. May be transient."

Signal: replication_lag_increasing
  -> Check: wal_growth_spike also firing?
    -> YES: Find query with highest WALBytes delta
      -> ROOT CAUSE: "WAL generation spiked: {rate}MB/s (was {prev_rate}MB/s).
         Top WAL contributor: query {queryid} ({wal_mb}MB/call). Replica cannot
         keep up with write volume."
    -> NO: Parse replica lag breakdown (write_lag, flush_lag, replay_lag)
      -> Identify bottleneck stage (largest lag component)
      -> ROOT CAUSE: "Replica {addr} bottleneck at {stage} stage.
         write_lag={w}, flush_lag={f}, replay_lag={r}. Check replica
         {resource} capacity."
         Where resource = "network" for write, "disk I/O" for flush,
         "CPU/apply" for replay.

Signal: vacuum_blocked
  -> Check: Locks[] contains any PID with State="idle in transaction"
            AND that PID's backend_xmin is the oldest active xmin?
    -> YES: ROOT CAUSE: "Transaction on PID {pid} (running {duration}) holds
             oldest xmin ({xmin}), preventing vacuum from cleaning {dead_count}
             dead tuples across {n} tables. Largest affected: {table}
             ({dead_pct}% dead tuples)."
    -> NO: Check Replication.Slots[] for any slot with RetainedBytes > 1GB
           AND Active=false?
      -> YES: ROOT CAUSE: "Inactive replication slot '{slot}' retaining
               {retained_gb}GB of WAL. Blocking vacuum on primary. Slot has
               been inactive — drop or reactivate."
      -> NO:  ROOT CAUSE: "Vacuum falling behind on {n} tables but no
               single blocker identified. Autovacuum workers may be
               insufficient. Current autovacuum_max_workers: {workers}."
```

Each decision tree leaf produces a deterministic `Incident` with `Source: "deterministic"` and `Confidence: 1.0`.

**Tier 2 — LLM Correlation** (covers the remaining ~30%):

Triggers when `>= rca.llm_correlation_threshold` (default: 3) signals fire simultaneously AND Tier 1 produces multiple incidents that don't share a common root PID or object. The LLM correlates them into a unified narrative.

**Execution model**: Tier 2 runs **asynchronously** in a background goroutine. The analyzer cycle is never blocked by LLM latency (which can be 10-30s). Flow:
1. Analyzer completes Tier 1 -> emits deterministic incidents immediately.
2. If correlation threshold met, enqueue a correlation request to a buffered channel (capacity 1; drop if full to avoid backlog).
3. Background goroutine picks up the request, calls LLM, merges the result into `sage.incidents`.
4. If the LLM response arrives after signals have already cleared (race), discard the result.

LLM System Prompt:

```

1. A list of active diagnostic findings from a PostgreSQL monitoring system
2. Key metric deltas between the current and previous monitoring snapshots
```

### 3.5 Deduplication Logic

An incoming incident is a **duplicate** of an existing open incident when ALL of:
1. Same `Source` (deterministic or llm)
2. Same set of `SignalIDs` (order-independent)
3. Same primary `AffectedObjects[0]` (the root cause object — e.g., same PID, same table)
4. Existing incident's `DetectedAt` is within `rca.dedup_window_minutes` (default 30)

On dedup match: bump `OccurrenceCount`, update `DetectedAt` to now, refresh `RootCause` text (metrics may have changed). Do NOT create a new row.

### 3.6 Auto-Resolution

An incident auto-resolves (sets `ResolvedAt = now()`) when:
- **For deterministic incidents**: The triggering signal(s) no longer fire for 2 consecutive analyzer cycles. "No longer fires" = the threshold condition in the signal definition table (Section 3.2) evaluates to false.
- **For LLM incidents**: ALL constituent `SignalIDs` stop firing for 2 consecutive cycles.

The 2-cycle requirement prevents flapping on transient recovery.

**Startup grace period**: On sidecar startup (or reconnect after a crash), the RCA engine enters a grace period of `rca.resolution_cycles + 1` cycles where it does NOT auto-resolve existing open incidents. This prevents mass false-resolution when the sidecar restarts and has no prior signal history. During the grace period, new incidents CAN still be detected.

### 3.7 Escalation

If an incident remains open (unresolved) for `rca.escalation_cycles` consecutive analyzer cycles (default 5), escalate:
- `severity` changes from `"warning"` to `"critical"`
- `EscalatedAt` is set
- A new alert is dispatched via `notify.EventDispatcher`

Already-critical incidents are not re-escalated.

## 4. Feature 2: Lock Chain Detection & Resolution

### 4.1 Problem

A migration acquires ACCESS EXCLUSIVE on a table where a long-running query is active. The migration waits. While it waits, every new query touching that table queues behind the migration. Within seconds, the connection pool is full. The operator sees "connections exhausted" but not the lock chain causing it.

### 4.2 Design

**Detection**: Run a lock chain query on `pg_locks` + `pg_stat_activity` every analyzer cycle. Build lock dependency chains. Identify the **root blocker** — the process at the head of the longest chain that is not itself waiting on a lock.

**Resolution**: Generate a `Finding` with `pg_terminate_backend()` or `pg_cancel_backend()` for the root blocker, gated by configurable policies and trust level.

### 4.3 Lock Chain Query

Uses `pg_blocking_pids()` (PG 9.6+) which natively handles relation locks, row-level locks (`transactionid`, `tuple`), and advisory locks. This avoids the pitfall of manually joining `pg_locks` on relation only, which would miss row-level contention (the most common lock contention pattern in OLTP workloads).

```sql
WITH RECURSIVE lock_chain AS (
    -- Base: every blocked process and its direct blockers
    SELECT
        sa.pid              AS blocked_pid,
        sa.query            AS blocked_query,
        sa.state            AS blocked_state,
        sa.query_start      AS blocked_since,
        sa.application_name AS blocked_app,
        blocker_pid,
        1                   AS depth,
        ARRAY[sa.pid, blocker_pid] AS chain
    FROM pg_stat_activity sa,
         LATERAL unnest(pg_blocking_pids(sa.pid)) AS blocker_pid
    WHERE sa.wait_event_type = 'Lock'

    UNION ALL

    -- Recursive: follow the chain upward through transitive blockers
    SELECT
        lc.blocked_pid,
        lc.blocked_query,
        lc.blocked_state,
        lc.blocked_since,
        lc.blocked_app,
        upper_blocker,
        lc.depth + 1,
        lc.chain || upper_blocker
    FROM lock_chain lc,
         LATERAL unnest(pg_blocking_pids(lc.blocker_pid)) AS upper_blocker
    WHERE upper_blocker != ALL(lc.chain)  -- Prevent cycles (deadlock)
      AND lc.depth < 10                   -- Safety: max chain depth
),
-- Identify root blockers: PIDs that block others but are not themselves blocked
root_blockers AS (
    SELECT DISTINCT blocker_pid AS root_pid
    FROM lock_chain
    WHERE blocker_pid NOT IN (SELECT blocked_pid FROM lock_chain)
)
SELECT
    rb.root_pid                      AS root_blocker_pid,
    sa.query                         AS root_blocker_query,
    sa.state                         AS root_blocker_state,
    sa.query_start                   AS root_blocker_since,
    sa.application_name              AS root_blocker_app,
    sa.wait_event_type               AS root_blocker_wait_type,
    max(lc.depth)                    AS chain_depth,
    array_agg(DISTINCT lc.blocked_pid) AS blocked_pids,
    count(DISTINCT lc.blocked_pid)   AS total_blocked
FROM root_blockers rb
JOIN lock_chain lc ON lc.blocker_pid = rb.root_pid
    OR (rb.root_pid = ANY(lc.chain) AND lc.blocked_pid != rb.root_pid)
JOIN pg_stat_activity sa ON sa.pid = rb.root_pid
GROUP BY rb.root_pid, sa.query, sa.state, sa.query_start,
         sa.application_name, sa.wait_event_type
ORDER BY total_blocked DESC;
```

**Locked relation detection**: Since `pg_blocking_pids()` doesn't directly report which relation is contended, we optionally enrich each chain with the locked relation by joining back to `pg_locks` for the root blocker PID:

```sql
-- Supplementary: find the contested relation for display purposes
SELECT DISTINCT l.relation::regclass::text AS locked_relation, l.mode
FROM pg_locks l
WHERE l.pid = $1  -- root_blocker_pid
  AND l.granted
  AND l.relation IS NOT NULL;
```

If no relation is found (e.g., row-level or advisory lock), `locked_relation` is set to `"row/advisory lock"` in the `LockChain` struct.

**Performance note**: `pg_blocking_pids()` executes in O(n) where n is the lock table size. On systems with > 10,000 concurrent locks, this query may take > 100ms. The query uses `statement_timeout` from `SafetyConfig.QueryTimeoutMs` as protection.

### 4.5 Safe Pattern Matching

The `safe_patterns` list is matched against `application_name` using **case-insensitive substring match**. A pattern "replication" matches application_name "walreceiver - replication" or "Replication Sender".

Additionally, pg_sage's own PID is always excluded (looked up once at startup via `pg_backend_pid()`).

Matching logic:

```go
func isSafeProcess(appName string, pid int, ownPID int, patterns []string) bool {
    if pid == ownPID {
        return true
    }
    lower := strings.ToLower(appName)
    for _, p := range patterns {
        if strings.Contains(lower, strings.ToLower(p)) {
            return true
        }
    }
    return false
}
```

### 4.6 Finding Generation

```go
func lockChainToFinding(chain LockChain, cfg LockChainConfig, ownPID int) *Finding {
    // Skip if root blocker is a safe process
    if isSafeProcess(chain.RootBlockerApp, chain.RootBlockerPID, ownPID, cfg.SafePatterns) {
        // Still emit an INFO finding (no SQL) so operators know
        return &Finding{
            Category:         "lock_chain",
            Severity:         "info",
            ObjectType:       "process",
            ObjectIdentifier: fmt.Sprintf("pid:%d", chain.RootBlockerPID),
            Title:            fmt.Sprintf("Lock chain: protected PID %d (%s) blocking %d sessions on %s",
                                  chain.RootBlockerPID, chain.RootBlockerApp,
                                  chain.TotalBlocked, chain.LockedRelation),
            Detail:           buildDetail(chain),
            Recommendation:   "Root blocker matches safe_patterns. Manual intervention required.",
            ActionRisk:       "", // Empty = no executor action
        }
    }

    severity := "warning"
    if chain.TotalBlocked >= cfg.CriticalBlockedThreshold {
        severity = "critical"
    }

    duration := time.Since(chain.RootBlockerSince)
    var risk, sql string

    switch {
    case chain.RootBlockerState == "idle in transaction" &&
         duration >= time.Duration(cfg.IdleInTxTerminateMinutes)*time.Minute:
        // Idle-in-tx has no graceful cancel — must terminate
        risk = "moderate"
        sql = fmt.Sprintf("SELECT pg_terminate_backend(%d); -- idle-in-tx for %s",
                  chain.RootBlockerPID, duration.Truncate(time.Second))

    case chain.RootBlockerState == "active" &&
         duration >= time.Duration(cfg.ActiveQueryCancelMinutes)*time.Minute:
        // Active query — try graceful cancel first
        risk = "safe"
        sql = fmt.Sprintf("SELECT pg_cancel_backend(%d); -- active for %s",
                  chain.RootBlockerPID, duration.Truncate(time.Second))

    default:
        // Below time thresholds — emit finding but no action
        risk = ""
        sql = ""
    }

    return &Finding{
        Category:         "lock_chain",
        Severity:         severity,
        ObjectType:       "process",
        ObjectIdentifier: fmt.Sprintf("pid:%d", chain.RootBlockerPID),
        Title:            fmt.Sprintf("Lock chain: PID %d blocking %d sessions on %s for %s",
                              chain.RootBlockerPID, chain.TotalBlocked,
                              chain.LockedRelation, duration.Truncate(time.Second)),
        Detail:           buildDetail(chain),
        Recommendation:   buildNarrative(chain, duration), // Human-readable explanation
        RecommendedSQL:   sql,
        RollbackSQL:      "", // Cannot undo a terminate/cancel
        ActionRisk:       risk,
    }
}
```

**ActionRisk behavior**: When `ActionRisk` is empty string, the executor skips the finding (no SQL to execute). The finding is still persisted for observability and feeds into the RCA engine.

### 4.7 Safety Rules

| Rule | Implementation | Rationale |
|------|---------------|-----------|
| Never terminate pg_sage's own PID | `isSafeProcess()` checks `ownPID` | Self-preservation |
| Never terminate replication senders | Default `safe_patterns: ["replication"]` | Prevents replica disconnection |
| Never terminate allowlisted apps | `isSafeProcess()` substring match | User-configured protection |
| Prefer `pg_cancel_backend` for active queries | Switch on `RootBlockerState` | Graceful cancellation first |
| Only `pg_terminate_backend` for idle-in-transaction | `idle in transaction` state check | No graceful option exists |
| Time threshold before action | Duration checked against config | Prevents premature termination |
| Minimum blocked count or chain depth | Finding only generated when `TotalBlocked >= min_blocked_threshold` | Avoid noise from single-victim locks |
| Trust-gated execution | `ActionRisk=moderate` requires autonomous + ramp day 31+; `ActionRisk=safe` requires advisory + ramp day 8+ | Existing executor trust model |

## 5. Feature 3: Storage Growth Forecasting

### 5.1 Problem

Disk full causes immediate PostgreSQL shutdown. No warning. WAL directories fill silently when replication slots lag or archive commands fail. Table bloat grows undetected until queries crawl and the disk is full.

### 5.2 Design

Track historical sizes at each collector cycle. Apply linear regression to project when storage thresholds will be breached. Separate forecasts for:

1. **Total database size** (`System.DBSizeBytes`)
2. **Per-table bloat** (dead tuple ratio trend from `Tables[]`)
3. **WAL/slot retention** (`Replication.Slots[].RetainedBytes`)

### 5.3 Data Collection

Record size data points from each snapshot. **No new collector fields needed** — all data is already in `Snapshot`.

```sql
-- Migration: v0.9.0-002
CREATE TABLE sage.size_history (
    id              BIGSERIAL PRIMARY KEY,
    collected_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    metric_type     TEXT NOT NULL CHECK (metric_type IN ('database', 'table', 'wal_slot')),
    object_name     TEXT NOT NULL,  -- db name, "schema.table", or slot name
    size_bytes      BIGINT NOT NULL,
    dead_tuple_pct  NUMERIC(5,2),   -- Only for metric_type='table'; NULL otherwise
    database_name   TEXT             -- Fleet mode: which database this belongs to
);

CREATE INDEX idx_size_history_lookup
    ON sage.size_history (metric_type, object_name, collected_at DESC);

-- Partial index for recent data (most queries look at last 7 days)
CREATE INDEX idx_size_history_recent
    ON sage.size_history (collected_at DESC)
    WHERE collected_at > now() - interval '7 days';
```

**Retention and downsampling** (managed by existing `internal/retention/` system):

| Age | Granularity | Strategy |
|-----|-------------|----------|
| 0-7 days | Every collector cycle (~1 min) | Raw data |
| 7-90 days | 1 row per hour | Keep the row closest to each hour boundary (no aggregation — a single representative point) |
| > 90 days | Deleted | Managed by `retention.snapshots_days` |

The downsampling job runs daily, deleting rows that fall between hourly boundaries for data older than 7 days. This avoids complex aggregation while reducing storage.

### 5.4 Disk Capacity Detection

How `disk_capacity_bytes` is determined when config value is 0 (auto-detect):

| Platform | Method |
|----------|--------|
| Self-managed | Query filesystem: `SELECT pg_tablespace_location('pg_default')` then `df` equivalent via `pg_stat_file` or OS-level check. Fallback: use `disk_capacity_bytes` config. |
| RDS / Aurora | `SELECT setting FROM pg_settings WHERE name = 'data_directory'` + `allocated_storage` from RDS API. **Practical fallback**: require `forecaster.disk_capacity_bytes` in config for managed services. Log a warning at startup if auto-detect fails and config is 0. |
| Cloud SQL | Same as RDS — require config or log warning. |

If auto-detection fails AND `disk_capacity_bytes = 0`, the forecaster skips absolute "days until full" calculations and only reports growth rate trends.

### 5.5 Forecasting Algorithm

```go
// internal/forecaster/growth.go

// LinearForecast predicts when a metric will cross a threshold.
type LinearForecast struct {
    Metric          string    `json:"metric"`           // "database", "table", "wal_slot"
    ObjectName      string    `json:"object_name"`
    CurrentBytes    int64     `json:"current_bytes"`
    GrowthRateBytes int64     `json:"growth_rate_bytes_per_day"` // Can be negative
    ThresholdBytes  int64     `json:"threshold_bytes"`  // 0 if capacity unknown
    DaysUntilFull   int       `json:"days_until_full"`  // -1 if shrinking or capacity unknown
    ProjectedDate   time.Time `json:"projected_date"`   // Zero value if capacity unknown
    R2              float64   `json:"r2"`               // Coefficient of determination
    DataPoints      int       `json:"data_points"`      // Number of points used
    DatabaseName    string    `json:"database_name,omitempty"`
}
```

Algorithm (ordinary least squares):

1. Query `sage.size_history` for the target metric over the last `forecaster.lookback_days` (existing config field, default 7).
2. Require at least `forecaster.min_data_points` rows (default 24). If insufficient, skip.
3. Compute OLS regression: `size_bytes = slope * unix_timestamp + intercept`.
4. Compute R^2. If `R2 < forecaster.min_r_squared` (default 0.5), flag as "irregular growth — forecast unreliable" and emit an INFO-level finding only.
5. If `slope <= 0` (shrinking or stable), set `DaysUntilFull = -1`. No alert.
6. If `ThresholdBytes > 0` (capacity known): `DaysUntilFull = (ThresholdBytes - CurrentBytes) / GrowthRateBytes`.
7. Generate findings at configured alert horizons.

### 5.6 Finding Generation

**Database size forecasts:**

| DaysUntilFull | Severity | Category | R^2 Requirement |
|---------------|----------|----------|-----------------|
| <= 3 | critical | `storage_forecast` | R^2 >= 0.8 (high confidence required for critical) |
| <= 7 | warning | `storage_forecast` | R^2 >= 0.5 |
| <= 30 | info | `storage_forecast` | R^2 >= 0.5 |
| > 30 or unknown capacity | No finding (data stored for API/dashboard) | — | — |

**Step-change resilience**: Linear regression is sensitive to sudden one-time step-changes (e.g., a migration adding 50GB). When R^2 < 0.5, the forecast is flagged as unreliable. For critical alerts, the R^2 >= 0.8 requirement provides additional protection against false positives from noisy data. Future improvement: detect step-changes via median absolute deviation (MAD) and segment the regression to points after the step.

**WAL slot retention** (checked every cycle, no forecast needed — it's a point-in-time check):

| Condition | Severity | Category |
|-----------|----------|----------|
| `RetainedBytes > disk_capacity * 0.50` | critical | `wal_slot_retention` |
| `RetainedBytes > disk_capacity * 0.25` | warning | `wal_slot_retention` |
| Slot inactive AND `RetainedBytes > 1GB` | warning | `wal_slot_retention` |

**Per-table bloat trend** (if dead_tuple_pct increasing > 5% absolute over lookback window):

| Condition | Severity | Category |
|-----------|----------|----------|
| dead_tuple_pct > 50% and rising | critical | `bloat_trend` |
| dead_tuple_pct > 25% and rising | warning | `bloat_trend` |

## 6. Feature 4: Runaway Query Termination

### 6.1 Problem

A single runaway query cascades into connection exhaustion and full outage. PostgreSQL's `statement_timeout` is a blunt instrument — it applies equally to a 10-second report and a 10-minute batch job. No intelligent, policy-based query management exists.

### 6.2 Design

A policy engine in the executor that evaluates active queries (from `Snapshot.Locks[]` and `ConfigData.ConnectionStates`) against configurable rules and escalates through warn -> cancel -> terminate.

### 6.3 Policy Rules

```go
// internal/executor/runaway.go

// RunawayPolicy defines when a query is considered runaway.
type RunawayPolicy struct {
    Name               string `yaml:"name"`
    // At least one of these conditions must be true (OR logic).
    // Set to 0 to disable a condition within a policy.
    MaxDurationMinutes int    `yaml:"max_duration_minutes"`
    MaxBlockedSessions int    `yaml:"max_blocked_sessions"`
    // Escalation timing (in analyzer cycles, not wall time).
    WarnCycles         int    `yaml:"warn_cycles"`   // Cycles at WARNED before CANCEL
    CancelCycles       int    `yaml:"cancel_cycles"` // Cycles at CANCELLED before TERMINATE
}
```

**Condition evaluation**: A query matches a policy if ANY non-zero condition is met:
- `MaxDurationMinutes > 0` AND `now() - query_start > MaxDurationMinutes` (from `pg_stat_activity.query_start` via `Snapshot.Locks[]`)
- `MaxBlockedSessions > 0` AND this PID appears as a root blocker in lock chain analysis (Feature 2) blocking >= MaxBlockedSessions

**Why no temp disk policy**: `pg_stat_statements` only updates counters when a query **completes**. An in-flight query writing 100GB of temp files shows zero delta until it finishes. Temp disk detection for active queries would require `pg_ls_dir('base/pgsql_tmp')` or OS-level monitoring, which is unreliable across platforms (especially managed services). Deferred to a future version when `pg_stat_progress_*` views provide better in-flight temp tracking.

**Relationship to `statement_timeout`**: PostgreSQL's native `statement_timeout` is a blunt global instrument. pg_sage's runaway termination provides **surgical, policy-based** protection: different thresholds per policy, gradual escalation (warn -> cancel -> terminate), and awareness of lock chain impact. Users should still set a generous `statement_timeout` as a safety net; pg_sage handles the nuanced cases.

### 6.4 Escalation State Machine

```
NORMAL -> WARNED -> CANCELLED -> TERMINATED

State transitions:
  NORMAL -> WARNED:
    Query matches at least one policy for 2+ consecutive cycles.
    (2-cycle minimum prevents killing transient spikes.)
    Action: Emit WARNING finding. No backend action.

  WARNED -> CANCELLED:
    Query still matches after policy.WarnCycles additional cycles.
    Action: Emit CRITICAL finding with pg_cancel_backend() SQL.
    Executor runs it if trust level permits.

  CANCELLED -> TERMINATED:
    pg_cancel_backend was issued but query still running after
    policy.CancelCycles additional cycles.
    Action: Emit CRITICAL finding with pg_terminate_backend() SQL.
    Executor runs it if trust level permits.

  Any state -> CLEARED:
    Query no longer appears in pg_stat_activity.
    Action: Remove from tracking map. Log resolution.
```

**Cycle counter**: The analyzer maintains a monotonically increasing `cycleNumber uint64` incremented each time the analyzer runs. `TrackedQuery` records the cycle number at each state transition.

```go
// TrackedQuery tracks a query through the escalation state machine.
// Held in memory only — runaway queries are transient.
type TrackedQuery struct {
    PID              int
    QueryStart       time.Time // From pg_stat_activity.query_start — used with PID as composite key
    QueryID          int64
    QueryText        string    // Truncated to 200 chars for logging
    AppName          string    // application_name for safe_patterns check
    FirstSeenCycle   uint64    // Cycle when first matched a policy
    MatchedPolicy    string    // Policy name that triggered
    State            string    // "warned", "cancelled", "terminated"
    WarnedAtCycle    uint64
    CancelledAtCycle uint64
}

// trackerKey uniquely identifies a query execution. PID alone is insufficient
// because PostgreSQL recycles PIDs when connections close and reopen.
// If query_start changes for the same PID, it's a different query.
type trackerKey struct {
    PID        int
    QueryStart time.Time
}

type RunawayTracker struct {
    mu       sync.Mutex
    tracked  map[trackerKey]*TrackedQuery
    cycle    uint64
}

// On each cycle, the tracker:
// 1. Scans pg_stat_activity for active queries.
// 2. For each (PID, query_start) pair still present: evaluate policies.
// 3. For tracked entries whose (PID, query_start) no longer appears: remove (query finished).
// 4. If a PID exists with a DIFFERENT query_start than tracked: remove old entry, start fresh.
```

### 6.5 Multiple Policy Resolution

When a query matches multiple policies, the policy with the **shortest total escalation time** wins (sum of `WarnCycles + CancelCycles`). This means the most aggressive policy determines the escalation timeline. The winning policy name is recorded in `TrackedQuery.MatchedPolicy`.

### 6.6 Safety Rules

| Rule | Implementation | Rationale |
|------|---------------|-----------|
| Never kill pg_sage's own connections | Check against `ownPID` (pg_backend_pid) | Self-preservation |
| Never kill replication senders | Merged `safe_patterns` from lock_chain + runaway config | Prevents replica disconnection |
| Never kill pg_dump / pg_basebackup | Default `safe_patterns` | Prevents backup corruption |
| 2-cycle minimum before WARNED | `FirstSeenCycle` must be >= 2 cycles ago | Prevents killing transient spikes |
| Log every action | All state transitions logged with PID, query, duration, policy, reason | Post-mortem audit trail |
| Trust-gated execution | `pg_cancel_backend` = `ActionRisk:"safe"`, `pg_terminate_backend` = `ActionRisk:"moderate"` | Respects existing trust ramp |

**Trust interaction**: Cancel (safe risk) executes in `advisory` mode at ramp day 8+. Terminate (moderate risk) requires `autonomous` mode at ramp day 31+ during maintenance window. In `observation` mode, findings are logged but no backend actions execute.

## 7. Feature 5: Natural Language EXPLAIN

### 7.1 Problem

Understanding `EXPLAIN ANALYZE` output requires deep expertise. Most developers don't read query plans. The DBA becomes a bottleneck answering "why is this slow?" questions.

### 7.2 Design

An LLM-powered EXPLAIN interpreter accessible via REST API. Takes a query text or a `queryid` from `pg_stat_statements` and returns a structured, plain-English explanation.

### 7.3 Flow

```
1. Client sends POST /api/explain with query text or query_id.
2. Input validation:
   a. If query_id provided: look up query text from latest Snapshot.Queries[]
      matching QueryID. If not found, return 404.
   b. If query text provided: syntax-check via pg_parse (or EXPLAIN without
      ANALYZE as a dry run).
3. Check sage.explain_cache for a hit (same query_hash, not expired).
   If hit: return cached result immediately.
4. Open a new connection (NOT from the pool — dedicated, to set session params):
   a. SET transaction_read_only = on;
   b. SET statement_timeout = {explain.timeout_ms};
   c. BEGIN;
   d. EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) {query};
   e. ROLLBACK;  -- Always rollback, even for SELECTs
5. Collect table stats for tables referenced in the plan (sizes, row counts,
   index info from latest Snapshot).
6. Send to LLM:
   - System prompt (Section 7.5)
   - User message: plan JSON + query text + table context
   - Max tokens: explain.max_tokens
7. Parse LLM response via stripToJSON().
8. Cache result in sage.explain_cache.
9. Return ExplainResult to client.
```

### 7.4 Handling Parameterized Queries

Queries from `pg_stat_statements` contain `$1`, `$2` parameter placeholders. These cannot be executed with `EXPLAIN ANALYZE`. Handling:

- If `plan_only: true` (or query contains `$N` placeholders): use `EXPLAIN (FORMAT JSON)` without `ANALYZE`. This gives the estimated plan without execution.
- If `plan_only: false` AND query has no parameters: use `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)`.
- If `plan_only: false` AND query has parameters: automatically downgrade to `EXPLAIN (FORMAT JSON)` and include a note in the response: `"note": "EXPLAIN without ANALYZE — query contains parameter placeholders. Provide literal values for actual execution statistics."`

The API also accepts an optional `params` field to supply parameter values for parameterized queries:

```go
type ExplainRequest struct {
    Query    string   `json:"query,omitempty"`
    QueryID  int64    `json:"query_id,omitempty"`
    PlanOnly bool     `json:"plan_only,omitempty"`
    Params   []string `json:"params,omitempty"` // Values for $1, $2, etc.
}
```

When `Params` is provided, the query is executed with `PREPARE` + `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) EXECUTE` to safely bind parameters without SQL injection risk.

```go
    NodeType    string   `json:"node_type"`             // e.g. "Seq Scan"

    Description string   `json:"description"`           // Plain English
    TimeMs      *float64 `json:"time_ms,omitempty"`     // nil if plan_only
    Rows        int64    `json:"rows"`                  // Actual or estimated

```

### 7.6 LLM System Prompt

```

the original query, and table statistics, produce a structured analysis.

Your audience is a developer who does NOT understand query plans. Use plain
English. When you must use a technical term, define it in parentheses.

For each plan node, explain:
- What it does in simple terms (e.g., "Scans every row of the orders table")
- Whether it's efficient or problematic, and why
- What would make it faster, if anything

Always check for and report these issues:
- Sequential scans on tables with > 10,000 rows (suggest specific index columns)
- Nested loops where the inner side produces > 1,000 rows (suggest hash or merge join)
- Sort operations with "Sort Method: external merge" (disk spill — suggest work_mem increase or index)
- Bitmap heap scans with "Exact Heap Blocks" much less than "Lossy Heap Blocks" (suggest narrower index)
- Row estimate divergence > 10x between estimated and actual (suggest ANALYZE on the table)
- Index scans with Filter conditions (partial index opportunity)

When comparing estimated vs actual rows, flag if estimated is > 10x off in either direction.

CRITICAL: Respond with ONLY a JSON object matching this schema:
{
  "summary": "1-2 sentence overview of what the query does and how it performs",

  "recommendations": ["specific actionable step 1", "specific actionable step 2"],
```

### 7.7 Safety

| Concern | Mitigation |
|---------|------------|
| Arbitrary query execution | `transaction_read_only = on` prevents all writes. ROLLBACK always issued. |
| DML in EXPLAIN ANALYZE | Read-only transaction rejects INSERT/UPDATE/DELETE/TRUNCATE at execution time. |
| Resource exhaustion | `statement_timeout` caps execution time. Dedicated connection (not from pool). |
| SQL injection via query text | For raw queries: passed directly to `EXPLAIN` (PostgreSQL parses it, not us). For parameterized: `PREPARE`/`EXECUTE` binds values safely. |
| LLM token budget | Counts against `llm.token_budget_daily`. If exhausted, return raw plan JSON with `"note": "LLM budget exhausted — raw plan only"`. |
| Cache poisoning | Cache key is `query_hash` (FNV-64a hash of normalized query text). TTL prevents stale results. |
| DDL/unsafe statements | Reject queries starting with CREATE, DROP, ALTER, TRUNCATE, GRANT, REVOKE, COPY, CLUSTER, REINDEX before sending to the database. Return 400 with message. These can have side effects even inside a read-only transaction depending on PG version and extensions. |
| Sensitive data in plans | Plans may contain literal values from queries. The explain endpoint should be gated by the same auth as the rest of the API (OAuth/token). |

### 7.8 Schema

```sql
-- Migration: v0.9.0-003
CREATE TABLE sage.explain_cache (
    query_hash  BIGINT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at  TIMESTAMPTZ NOT NULL,
    plan_json   JSONB NOT NULL,
    explanation JSONB NOT NULL,
    database_name TEXT,
    PRIMARY KEY (query_hash, database_name)
);

-- TTL cleanup: retention system deletes rows where expires_at < now()
```

Cache key: `FNV-64a(normalized_query_text + sorted_params_string)` where normalization strips comments and collapses whitespace. When `params` is provided, the sorted params string is appended before hashing — different parameter values can produce different plans (e.g., `status = 'active'` may seq scan while `status = 'banned'` may index scan). When `plan_only = true` (no params), the cache key uses only the query text. In fleet mode, the primary key includes `database_name` since the same query can have different plans on different databases.

### 7.11 Testing Strategy

| Category | Test |
|----------|------|
| Happy path | SELECT with seq scan on 1M-row table -> explanation identifies missing index, suggests CREATE INDEX |
| Complex join | 4-table join with nested loop -> explanation identifies suboptimal join strategy |
| Already optimal | PK lookup -> explanation says "efficient index scan" |
| DML blocked | INSERT with plan_only=false -> read-only transaction rejects, returns error |
| Timeout | EXPLAIN ANALYZE on very slow query -> statement_timeout fires, returns error with message |
| Cache hit | Same query twice within TTL -> second call returns cached result with cached_at set |
| Cache miss | Same query after TTL expires -> fresh EXPLAIN + LLM call |
| LLM failure | LLM unavailable -> return raw plan JSON with note |
| LLM budget exhausted | Daily budget exceeded -> return raw plan JSON with note |
| Query ID | Valid pg_stat_statements queryid -> resolves to query text, explains |
| Query ID not found | Invalid queryid -> 404 with message |
| Parameterized query | Query with $1, $2 -> auto-downgrades to plan_only, includes note |
| Parameterized with values | Query with $1 + params: ["foo"] -> PREPARE/EXECUTE, full ANALYZE |
| Invalid query | Syntax error -> meaningful error before LLM call |
| Row estimate divergence | Plan with 100 estimated, 100000 actual -> warning about stale stats |
| Fleet mode | Explain request includes database context, cache keyed by (hash, database_name) |

## 13. Prerequisites & Operational Notes

### Connection Reservation

When `TotalBackends / MaxConnections > 0.95`, pg_sage itself may be unable to connect to run diagnostics (including the lock chain query that would identify the cause). Mitigations:

1. **Documentation**: Recommend users set `superuser_reserved_connections >= 3` (PG default is 3) and configure pg_sage's connection user as a superuser or use a dedicated connection slot.
2. **pg_sage pool sizing**: `PostgresConfig.MaxConnections` (pg_sage's pool, not PostgreSQL's) should be set low (default 3). Ensure the pool can connect even under saturation.
3. **Graceful degradation**: If pg_sage cannot connect, log the error with context ("cannot run lock chain detection — database connection limit reached") and retry on next cycle. Do not crash.

### PostgreSQL Version Requirements

| Feature | Minimum PG Version | Reason |
|---------|-------------------|--------|
| Lock Chain Detection | 9.6 | `pg_blocking_pids()` |
| All features | 12 | `pg_stat_statements` with `queryid`, `pg_ls_dir` permissions |
| WAL bytes tracking | 13 | `pg_stat_statements.wal_bytes` |
| IO stats for RCA | 16 | `pg_stat_io` |

Features degrade gracefully on older versions — signals that require missing views are simply not evaluated.
