# pg_sage v0.9.1 Spec: Log-Based RCA (Self-Managed)

> **Status**: Draft v3 (research + edge cases + Gemini 2.5 Pro review integrated)
> **Date**: 2026-04-13
> **Author**: jmass + Claude
> **Depends on**: v0.9 (RCA Engine, Signal Detection)
> **Theme**: Extend RCA with high-confidence signals from PostgreSQL log events.

---

## 1. Goals

1. Ingest PostgreSQL log files on self-managed instances and convert log events
   into RCA signals.
2. Surface confirmed incidents (deadlocks, OOM, crashes, disk full, connection
   refused) that snapshot-based polling cannot detect.
3. Correlate log-derived signals with existing metric-based signals in the
   decision tree and Tier 2 pipeline.
4. Keep the log ingestion backend pluggable so v0.10 can add cloud-provider
   backends without changing the signal layer.

### Non-Goals (v0.9.1)

- No managed-service log backends (CloudWatch, Cloud Logging, Azure Monitor).
  Those are v0.10.
- No `stderr` format parsing. Only `jsonlog` (PG15+) and `csvlog`.
- No log retention, search, or storage. pg_sage reads logs; it does not become
  a log aggregator.
- No new REST endpoints. Log signals feed into the existing RCA and incident
  APIs.
- No historical log analysis. Only real-time events after the sidecar starts.
- No internationalized message matching. Substring-based patterns are
  English-locale-only in v0.9.1.

## 2. Architecture

```
┌─────────────────────────────────────────────────┐
│  PostgreSQL Instance (self-managed)             │
│  log_destination = 'jsonlog' or 'csvlog'        │
│  logging_collector = on                         │
│  log_directory = '/var/log/postgresql'           │
└────────────────────┬────────────────────────────┘
                     │ log files
                     ▼
┌─────────────────────────────────────────────────┐
│  LogWatcher  (internal/logwatch/)               │
│                                                 │
│  ┌──────────┐   ┌──────────┐   ┌────────────┐  │
│  │ Tailer   │──▶│ Parser   │──▶│ Classifier │  │
│  │ (fsnotify│   │ (jsonlog │   │ (pattern   │  │
│  │  + poll) │   │  /csvlog)│   │  matcher)  │  │
│  └──────────┘   └──────────┘   └─────┬──────┘  │
│                                      │          │
│                               []*rca.Signal     │
└──────────────────────────────────────┬──────────┘
                                       │
                                       ▼
┌─────────────────────────────────────────────────┐
│  RCA Engine  (internal/rca/)                    │
│                                                 │
│  Analyze() calls logSource.Drain() internally   │
│       │                                         │
│  detectSignals()  ◀── metric signals (existing) │
│       +                                         │
│  Drain()          ◀── log signals               │
│       │                                         │
│  Combined signal list feeds:                    │
│  ├── Decision Trees (Tier 1)                    │
│  └── LLM Correlation (Tier 2)                   │
└─────────────────────────────────────────────────┘
```

### 2.1 Component Summary

| Component | Package | Responsibility |
|-----------|---------|----------------|
| **Tailer** | `internal/logwatch/tailer.go` | Follow log files via fsnotify on the directory; handle rotation (new file CREATE event) and external truncation (copytruncate). Fallback to poll-based tailing when fsnotify unavailable. |
| **Parser** | `internal/logwatch/parser.go` | Parse jsonlog (JSON per line) and csvlog (RFC 4180 via `encoding/csv`). Pre-filter: skip LOG-level lines that don't contain signal-relevant keywords. Emit structured `LogEntry`. |
| **Classifier** | `internal/logwatch/classifier.go` | Match `LogEntry` fields to signal IDs. Primary match on SQLState code (locale-independent); fallback to case-insensitive substring match on Message for signals without unique SQLState. Dedup and rate-limit. |
| **LogWatcher** | `internal/logwatch/watcher.go` | Top-level orchestrator implementing `LogSource` interface. Owns the Tailer→Parser→Classifier pipeline. Exposes `Drain() []*rca.Signal`. |

### 2.2 Package Boundary

`Drain()` returns `[]*rca.Signal` — the same type used by metric detectors.
There is no `LogSignal` type. Log-specific metadata (PID, SQLState, raw
message) is stored in the `Metrics` map:

```go
Metrics["source_pid"]   int
Metrics["sql_state"]    string
Metrics["raw_message"]  string  // truncated to maxRawMessageLen (500)
Metrics["database"]     string
Metrics["application"]  string
```

This avoids circular imports and keeps the RCA engine agnostic to signal
origin.

### 2.3 Drain() Lifecycle

1. `Drain()` is **destructive**: returns accumulated signals and clears the
   internal buffer.
2. `rca.Engine` gains a `SetLogSource(LogSource)` method.
3. `Analyze()` calls `e.logSource.Drain()` internally at the start of each
   cycle, before `detectSignals()`, and merges the results.
4. Internal buffer capacity: 10,000 signals. On overflow, oldest signals are
   dropped and a warning is logged.
5. Log signals participate in Tier 2 (LLM correlation) like any other signal.
   They are included in `findUncoveredSignals()` if not consumed by a Tier 1
   tree.

### 2.4 "Same Cycle" Definition

Log signals drained during an `Analyze()` call are considered part of the
current cycle regardless of their embedded `FiredAt` timestamp. The `FiredAt`
timestamp (from the log file) is for display/investigation only — it does NOT
affect whether they are considered "in the same cycle" as metric signals.

### 2.5 Fleet Mode

One `LogWatcher` per PostgreSQL cluster (host:port), not per database.
Log entries contain `database_name`, so signals include
`Metrics["database"]` for routing. In fleet mode, the main loop deduplicates
LogWatcher instances by resolved `log_directory` path. Each per-database RCA
engine receives only signals matching its database name (or signals with
empty database name, like background worker events).

## 3. Data Structures

### 3.1 LogEntry (parsed log line)

```go
// LogEntry is a single parsed PostgreSQL log line.
type LogEntry struct {
    Timestamp    time.Time // log_time (always UTC after parsing)
    PID          int       // process_id
    SessionID    string    // session_id
    Database     string    // database_name
    User         string    // user_name
    ErrorLevel   string    // error_severity: LOG, WARNING, ERROR, FATAL, PANIC
    SQLState     string    // sql_state_code (e.g. "40P01" = deadlock)
    Message      string    // message
    Detail       string    // detail
    Hint         string    // hint
    Query        string    // query (if available)
    Application  string    // application_name
}
```

**Timestamp handling**: All timestamps are parsed with their embedded timezone
info (jsonlog includes UTC offset; csvlog includes offset in PG14+). After
parsing, converted to UTC immediately. `FiredAt` on the emitted Signal is set
from `LogEntry.Timestamp`, not `time.Now()`.

**Encoding**: Non-UTF-8 bytes in log lines are replaced with U+FFFD via
`strings.ToValidUTF8()` before parsing. This handles SQL_ASCII databases.

### 3.2 LogSource Interface (for v0.10 pluggability)

```go
// LogSource provides log-derived signals to the RCA engine.
type LogSource interface {
    Start(ctx context.Context) error
    Drain() []*rca.Signal
    Stop()
}
```

`FileWatcher` (v0.9.1) and each cloud poller (v0.10) implement this
interface. The RCA engine consumes signals through `Drain()` without knowing
which backend produced them.

### 3.3 Config

```go
// LogWatchConfig controls PostgreSQL log file monitoring (v0.9.1).
type LogWatchConfig struct {
    Enabled              bool     `yaml:"enabled"`
    LogDirectory         string   `yaml:"log_directory"`
    Format               string   `yaml:"format"`
    PollIntervalMs       int      `yaml:"poll_interval_ms"`

    MaxLineLenBytes      int      `yaml:"max_line_len_bytes"`
    TempFileMinBytes     int      `yaml:"temp_file_min_bytes"`
    MaxLinesPerCycle     int      `yaml:"max_lines_per_cycle"`
    ExcludeApplications  []string `yaml:"exclude_applications"`
    SlowQueryEnabled     bool     `yaml:"slow_query_enabled"`
}
```

Config keys (10 total):
```
logwatch.enabled                   bool
logwatch.log_directory             string
logwatch.format                    string     ("jsonlog" or "csvlog")
logwatch.poll_interval_ms          int_pos    (default: 1000)
logwatch.dedup_window_seconds      int_pos    (default: 60)

logwatch.max_lines_per_cycle       int_pos    (default: 10000)
logwatch.exclude_applications      string     (comma-separated, default: "pg_sage")
logwatch.slow_query_enabled        bool       (default: false, opt-in)
```

**Prerequisite**: `logwatch` requires `rca.enabled = true`. If RCA is
disabled, logwatch is automatically disabled regardless of
`logwatch.enabled`.

## 4. Log Signal Definitions

### 4.1 Signal Catalog (18 signals)

#### P0 — Must-have (database-down or data-loss events)

| Signal ID | Fires When | SQLState | ErrorLevel | Severity | Match Strategy |
|-----------|-----------|----------|------------|----------|----------------|
| `log_deadlock_detected` | Deadlock resolved by PG | `40P01` | ERROR | critical | SQLState |
| `log_connection_refused` | Max connections reached | `53300` | FATAL | critical | SQLState |
| `log_out_of_memory` | Shared memory or work_mem exhaustion | `53200` | ERROR | critical | SQLState |
| `log_disk_full` | No space left on device | `53100` | ERROR/PANIC | critical | SQLState |
| `log_panic_server_crash` | Server crash / PANIC | — | PANIC | critical | ErrorLevel=PANIC OR substring "server process was terminated by signal" |
| `log_data_corruption` | Checksum failure / invalid page | `XX001`/`XX002` | WARNING/ERROR | critical | SQLState prefix `XX` |
| `log_txid_wraparound_warning` | XID wraparound approaching | — | WARNING | critical | Substring "must be vacuumed within" |
| `log_archive_failed` | WAL archive command failed | — | LOG | critical | Substring "archive command failed" |
| `log_temp_file_created` | Temp file > threshold created | — | LOG | warning | Substring "temporary file" AND parsed size >= `temp_file_min_bytes` |
| `log_checkpoint_too_frequent` | Checkpoints too frequent | — | LOG | warning | Substring "checkpoints are occurring too frequently" |

#### P1 — Strong candidates

| Signal ID | Fires When | SQLState | ErrorLevel | Severity | Match Strategy |
|-----------|-----------|----------|------------|----------|----------------|
| `log_lock_timeout` | lock_timeout exceeded | `55P03` | ERROR | warning | SQLState |
| `log_statement_timeout` | statement_timeout exceeded | `57014` | ERROR | warning | SQLState |
| `log_replication_conflict` | Query canceled on standby | `40001` | ERROR/FATAL | warning | Substring "conflict with recovery" |
| `log_wal_segment_removed` | Replica WAL gap | — | ERROR/FATAL | critical | Substring "WAL segment has already been removed" |
| `log_autovacuum_cancel` | Autovacuum canceled by lock | — | LOG | warning | Substring "canceling autovacuum task" |
| `log_replication_slot_inactive` | Slot lag warning | — | WARNING | warning | Substring "replication slot" AND "inactive" |
| `log_authentication_failure` | Auth failures | `28P01`/`28000` | FATAL | warning | SQLState prefix `28` |
| `log_slow_query` | Duration exceeds `log_min_duration_statement` | — | LOG | info | Substring "duration:" AND parsed duration > 0. **Opt-in**: requires `logwatch.slow_query_enabled: true` (default: false). |

### 4.2 Pattern Matching Strategy

**Primary**: Match on `SQLState` code where available. SQLState is
locale-independent and unambiguous. 10 of 17 signals have unique SQLState
codes.

**Fallback**: For 7 signals without unique SQLState (`log_temp_file_created`,
`log_checkpoint_too_frequent`, `log_archive_failed`,
`log_replication_slot_inactive`, `log_autovacuum_cancel`,
`log_panic_server_crash`, `log_txid_wraparound_warning`,
`log_wal_segment_removed`, `log_replication_conflict`), use case-insensitive
substring match on the English `Message` field. These signals are
English-locale-only in v0.9.1.

**Self-exclusion**: Entries where `Application` matches any value in
`logwatch.exclude_applications` (default: `["pg_sage"]`) are excluded from
`log_statement_timeout`, `log_lock_timeout`, and `log_deadlock_detected`
signals. For self-inflicted deadlocks, a signal IS emitted but with
`Metrics["self_inflicted"] = true` and the decision tree notes it as
internal.

### 4.3 Dedup Semantics

**Dedup key**: `(signal_id, database)`. Same signal from the same database
within `dedup_window_seconds` is collapsed into one signal.

**Counter**: The dedup entry tracks `occurrence_count` and `affected_pids`
(set of unique PIDs, capped at 20). The emitted signal includes:
```go
Metrics["occurrence_count"] int     // how many raw events were collapsed
Metrics["affected_pids"]    []int   // unique PIDs (max 20)
```

**Rate limiting**: After `max_lines_per_cycle` (default 10,000) lines are
parsed in a single Drain interval, the parser stops reading and emits a
synthetic `log_watcher_overloaded` warning signal. This prevents CPU
exhaustion when `log_statement = 'all'` is set.

### 4.4 Signal-to-Metrics Mapping

Each log signal populates its `Metrics` map with common fields plus
event-specific data:

```go
// Common fields on all log signals:
Metrics["source_pid"]        int
Metrics["sql_state"]         string
Metrics["raw_message"]       string  // truncated, max 500 chars
Metrics["database"]          string
Metrics["application"]       string
Metrics["occurrence_count"]  int
Metrics["affected_pids"]     []int

// log_deadlock_detected — additional:
Metrics["query"]    string  // truncated, max 200 chars
Metrics["detail"]   string  // deadlock graph from PG

// log_temp_file_created — additional:
Metrics["size_bytes"]  int64   // parsed from message
Metrics["path"]        string  // temp file path

// log_disk_full — additional:
Metrics["target_path"]  string  // which filesystem/file was full

// log_panic_server_crash — additional:
Metrics["signal_number"]  int   // if terminated by signal

// log_data_corruption — additional:
Metrics["block_number"]  int    // if available
Metrics["relation_path"] string // OID path to the relation

// log_authentication_failure — additional:
Metrics["user"]     string
Metrics["source_ip"] string  // if available from log entry
```

**Constants** (not configurable):
```go
const maxRawMessageLen = 500
const maxQueryLen = 200
const maxAffectedPIDs = 20
```

## 5. Decision Tree Integration

### 5.1 Enhanced Existing Trees

Log signals strengthen existing decision trees with high-confidence branches:

```
Signal: connections_high
  -> NEW BRANCH (checked first):
     Check: log_connection_refused fired in same cycle?
       -> YES: ROOT CAUSE: "Connection limit hit — PostgreSQL actively
                refusing connections. {n} connection_refused events in
                last cycle. Application errors likely occurring."
                Severity: critical, Confidence: 1.0
  -> (existing branches follow)

Signal: vacuum_blocked
  -> NEW BRANCH (checked first):
     Check: log_autovacuum_cancel fired in same cycle?
       -> YES: Check occurrence_count in dedup window
         -> > 3: ROOT CAUSE: "Autovacuum repeatedly canceled by lock
                  conflicts ({n} cancellations). Dead tuples accumulating.
                  Consider adjusting autovacuum_vacuum_cost_delay or
                  scheduling maintenance windows."
                  Severity: warning, Confidence: 0.95
     Check: log_txid_wraparound_warning also firing?
       -> YES: ROOT CAUSE: "EMERGENCY: Autovacuum blocked while XID
                wraparound approaches. Immediate manual VACUUM FREEZE
                required. Database will shut down if wraparound completes."
                Severity: critical, Confidence: 1.0
  -> (existing branches follow)
```

### 5.2 New Decision Trees (Log-Only)

```
Signal: log_deadlock_detected
  -> Check: self_inflicted == true?
    -> YES: ROOT CAUSE: "Deadlock involved pg_sage's own session.
             Likely schema migration conflict or concurrent sidecar
             restart. No application action needed."
             Severity: info, Confidence: 1.0
  -> Check: occurrence_count > 1 in dedup window?
    -> YES: ROOT CAUSE: "Recurring deadlock on database {db}. {n}
             occurrences in {window}. Likely application-level lock
             ordering issue. Queries: {query1}, {query2}."
             Severity: critical, Confidence: 1.0
    -> NO: ROOT CAUSE: "Single deadlock detected. PID {pid} was the
            victim. Query: {query}. Review application retry logic."
            Severity: warning, Confidence: 1.0

Signal: log_out_of_memory
  -> Check: log_temp_file_created also fired in same cycle?
    -> YES: ROOT CAUSE: "Memory exhaustion from sort/hash operations
             spilling to disk. Largest temp file: {size}MB. Consider
             increasing work_mem or adding indexes to avoid large sorts."
             Severity: critical, Confidence: 0.9
    -> NO: ROOT CAUSE: "Out of shared memory. Check shared_buffers,
            max_locks_per_transaction, or max_pred_locks_per_transaction."
            Severity: critical, Confidence: 0.9

Signal: log_panic_server_crash
  -> Check: log_out_of_memory fired in same cycle?
    -> YES: ROOT CAUSE: "OOM killer terminated PostgreSQL. Check
             vm.overcommit_memory, shared_buffers sizing, and system
             memory pressure. Review dmesg for OOM events."
             Severity: critical, Confidence: 0.95
  -> Check: log_disk_full fired in same cycle?
    -> YES: ROOT CAUSE: "Server crash due to disk exhaustion.
             Identify which filesystem is full (data, WAL, or temp)."
             Severity: critical, Confidence: 0.95
  -> DEFAULT: ROOT CAUSE: "PostgreSQL server crash detected (PANIC).
              All sessions terminated, crash recovery required.
              Check system logs (dmesg, syslog) for hardware errors
              or extension bugs."
              Severity: critical, Confidence: 1.0

Signal: log_disk_full
  -> Check: log_archive_failed also firing?
    -> YES: ROOT CAUSE: "WAL archive failure caused disk exhaustion
             in pg_wal. Unarchived segments are accumulating. Fix
             archive_command, then clear the backlog."
             Severity: critical, Confidence: 1.0
             RecommendedSQL: "SELECT * FROM pg_stat_archiver;"
    -> NO: ROOT CAUSE: "Disk full: {target_path}. Identify largest
            consumers (tables, indexes, WAL, temp files) and free space
            immediately. PostgreSQL may refuse writes or crash."
            Severity: critical, Confidence: 1.0

Signal: log_archive_failed (standalone — no disk_full co-signal)
  -> ROOT CAUSE: "WAL archive command failing. Unarchived WAL will
     accumulate until pg_wal fills the volume. Investigate
     archive_command and destination storage immediately."
     Severity: critical, Confidence: 1.0
     RecommendedSQL: "SELECT * FROM pg_stat_archiver;"
     StickyUntilCleared: true (see §5.3)

Signal: log_data_corruption
  -> ROOT CAUSE: "DATA CORRUPTION DETECTED: {message}. Affected
     relation: {relation_path}, block: {block_number}. Immediate
     action: identify affected table/index, check for hardware issues
     (disk, memory), prepare backup restore."
     Severity: critical, Confidence: 1.0

Signal: log_txid_wraparound_warning (standalone — no vacuum_blocked)
  -> ROOT CAUSE: "Transaction ID wraparound warning for database
     {db}. Must be vacuumed within {n} transactions. Run VACUUM
     FREEZE immediately. If ignored, PostgreSQL will shut down
     to prevent data loss."
     Severity: critical, Confidence: 1.0
     RecommendedSQL: "VACUUM FREEZE;"

Signal: log_wal_segment_removed
  -> ROOT CAUSE: "Replica WAL gap: requested segment already removed.
     Replica cannot recover without a full base backup rebuild.
     Investigate replica downtime cause, increase wal_keep_size or
     use replication slots."
     Severity: critical, Confidence: 1.0

Signal: log_authentication_failure
  -> Check: occurrence_count > 20 in dedup window?
    -> YES: Check distinct source IPs in affected_pids/metrics
      -> Many distinct IPs (> 5): ROOT CAUSE: "Possible brute-force
               attack: {n} auth failures from {ip_count} distinct sources
               in {window}. Review fail2ban and connection rate limiting."
               Severity: critical, Confidence: 0.9
      -> Single/few IPs: ROOT CAUSE: "Mass auth failure from {ip}:
               {n} failures in {window}. Likely misconfigured application
               deployment or credential rotation issue."
               Severity: critical, Confidence: 0.9
  -> Check: occurrence_count > 5 in dedup window?
    -> YES: ROOT CAUSE: "Elevated authentication failures: {n} in
             {window}. Check recent pg_hba.conf or credential changes."
             Severity: warning, Confidence: 0.8
    -> NO: ROOT CAUSE: "Authentication failure for user {user} on
            database {db}. Check pg_hba.conf and connection strings."
            Severity: info, Confidence: 0.7
```

### 5.3 Auto-Resolve Behavior for Log Signals

Most log signals are ephemeral: a deadlock fires once and the incident
auto-resolves after `resolution_cycles` clear cycles. This is correct.

**Sticky signals**: `log_archive_failed` does NOT auto-resolve via the normal
cycle-based mechanism. Instead, the incident persists until one of:
- A `log_archive_success` pattern is seen in logs (LOG: "archived
  transaction log file")
- Manual resolution via the API

Implementation: sticky signals set a high `ResolutionCycles` override (e.g.,
999) on their incident, effectively disabling auto-resolve.

### 5.4 Incident DatabaseName

Log-based incidents set `DatabaseName` from `Metrics["database"]`. If the log
entry's database field is empty (background worker logs), use the sidecar's
configured database name.

## 6. Tailer Behavior

### 6.1 Startup (Cold Start)

1. Resolve `log_directory`: if relative (no leading `/` on Linux, no drive
   letter on Windows), join with `data_directory` via `filepath.Join()`.
   Resolve symlinks via `filepath.EvalSymlinks()`.
2. List files in directory. Sort by modification time descending.
3. Open ONLY the most recent file. **Seek to (END - 1MB)** or the start of
   the file, whichever is closer. This "lookback" captures recent critical
   events (e.g., a PANIC that caused the sidecar to restart) without
   reprocessing the entire log history. Lookback lines with timestamps older
   than `dedup_window_seconds` are discarded. Lookback lines within the
   dedup window are processed normally — the dedup mechanism prevents
   duplicate signals if the sidecar restarted quickly.
4. Start fsnotify watch on the directory.

Design decision: Full historical log analysis is out of scope. The 1MB
lookback is a pragmatic compromise: it catches the crash that triggered a
restart without processing megabytes of old logs.

### 6.2 Log Rotation

**PostgreSQL internal rotation** (`log_rotation_age`, `log_rotation_size`):
PG creates a new file. fsnotify fires a CREATE event. The tailer finishes
reading the old file (drain remaining bytes), then switches to the new file.

**External rotation** (`logrotate` with `copytruncate`): The current file is
truncated to 0 bytes without creating a new one. The tailer detects this:
if a read returns EOF and the file size is less than the last-read offset,
reset offset to 0 and resume reading from the beginning.

**Active file detection**: Primary method: query `SHOW log_filename` to
get the strftime pattern, expand it for the current time, and look for that
file. Fallback (PG unreachable): filter files by expected extension
(`.log`, `.csv`, `.json` from the configured format), sort by mtime, take
the latest. Log a warning when using the mtime fallback. Ignore files with
future mtime.

### 6.3 Poll Fallback

If fsnotify initialization fails (common on NFS, CIFS, some container
mounts), fall back to polling at `poll_interval_ms`. Log a warning on
fallback. The poll loop: stat the file, if size increased read new bytes,
check for new files by mtime.

### 6.4 Permissions

On startup, attempt to open the current log file for reading. If permission
denied, log:
```
Cannot read log file {path}: permission denied.
Linux: ensure sidecar user is in the postgres group, or set
  log_file_mode = 0640 in postgresql.conf.
Windows: grant Read permission on the log directory to the
  sidecar service account.
```
Gracefully disable logwatch. Do not fail sidecar startup.

## 7. Parser Behavior

### 7.1 Pre-Filtering (High Volume)

With `log_statement = 'all'`, a busy database generates thousands of
LOG-level lines per second. The parser pre-filters to avoid full struct
allocation for 99%+ of lines:

**jsonlog**: Check `"error_severity"` field before full JSON unmarshal.
Lines with severity `LOG` that don't contain any signal-relevant keyword
are discarded without full parsing.

**csvlog**: Check column 12 (error_severity, 0-indexed 11) before full
CSV parsing.

**Keywords for LOG-level pre-filter**: `deadlock`, `temporary file`,
`checkpoint`, `archive`, `autovacuum`, `replication slot`, `vacuum`,
`terminated by signal`.

Lines with severity WARNING, ERROR, FATAL, or PANIC are always fully parsed.

### 7.2 csvlog Multi-Line Messages

Use Go's `encoding/csv` reader which handles RFC 4180 quoted fields
natively, including embedded newlines. Set `csv.Reader.LazyQuotes = true`
for robustness. Discard any single CSV record exceeding `max_line_len_bytes`.
If a file read returns no data for 5 seconds while a partial record is
buffered, emit a warning and discard the partial record.

### 7.3 csvlog Column Compatibility

PG14 added the `query_id` column (column 27) to csvlog. Earlier versions
have 26 columns. The parser handles both: parse by known column positions
up to the columns that exist. Missing columns are empty strings. Warn if
fewer than 23 columns (minimum for meaningful data).

### 7.4 jsonlog Notes

Neither jsonlog nor csvlog are affected by `log_line_prefix`. The parser
ignores this setting. jsonlog always outputs a fixed set of JSON fields.

## 8. Auto-Detection

On startup, if `logwatch.enabled` is true and `log_directory` / `format` are
blank, the sidecar queries:

```sql
SHOW log_destination;    -- check for 'jsonlog' or 'csvlog'
SHOW log_directory;      -- absolute or relative to data_directory
SHOW data_directory;     -- resolve relative log_directory
SHOW logging_collector;  -- must be 'on'
```

**State machine**:
```
pending_autodetect → active             (success)
pending_autodetect → disabled_no_format (no jsonlog/csvlog in log_destination)
pending_autodetect → disabled_no_perms  (file permission denied)
pending_autodetect → pending_autodetect (PG down, retry next cycle)
```

If `logging_collector = off`, disable with: "logwatch requires
`logging_collector = on`. Set in postgresql.conf and restart."

If `log_destination` doesn't include `jsonlog` or `csvlog`, disable with
warning. Don't fail sidecar startup.

**Retry**: If PG is down at startup, auto-detection stays in
`pending_autodetect` and retries on the next analyzer cycle.

**Relative paths**: If `log_directory` doesn't start with `/` (Linux) or
doesn't contain `:` (Windows drive letter), resolve relative to
`data_directory` via `filepath.Join()`. Normalize with `filepath.Clean()`.

## 11. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Log file permissions | Can't read logs | Startup permission check, clear error message, graceful disable |
| High-volume logs (log_statement=all) | CPU spike | Pre-filter by severity + keyword, max_lines_per_cycle cap, rate limiter |
| Log rotation missed (copytruncate) | Stale FD, missed events | Detect truncation (file size < offset), reset to 0 |
| csvlog multi-line messages | Partial parse | Use encoding/csv with LazyQuotes, timeout on incomplete records |
| jsonlog not available (PG < 15) | Format unsupported | Fall back to csvlog. If neither, disable with warning |
| Windows path handling | Slash differences | `filepath.Join()`, `filepath.EvalSymlinks()`, Windows-specific tests |
| PG down at startup | Auto-detect fails | Retry on next analyzer cycle (pending_autodetect state) |
| SELinux blocking reads | Permission denied | Document: check `ausearch -m avc -ts recent` |
| Self-inflicted signals | Noise from pg_sage's own queries | `exclude_applications` config, `self_inflicted` metric flag |
| Non-English locale | Substring patterns fail | Primary match on SQLState (10/17 signals); document English-only fallback |

## 13. Self-Action Correlation

pg_sage is an autonomous agent that executes DDL/DML. When those actions cause
problems, the RCA engine should identify pg_sage as a contributor. This builds
operator trust: an agent that says "I may have caused this" is more trustworthy
than one that silently ignores its own impact.

### 13.1 Action Context Enrichment

Every RCA incident gains a new field:

```go
type Incident struct {
    // ... existing fields ...
    RecentSageActions []ActionContext `json:"recent_sage_actions,omitempty"`
}

type ActionContext struct {
    ActionID    int64     `json:"action_id"`
    ExecutedAt  time.Time `json:"executed_at"`
    ActionType  string    `json:"action_type"`
    SQLExecuted string    `json:"sql_executed"`
    RollbackSQL string    `json:"rollback_sql,omitempty"`
}
```

At the end of each `Analyze()` cycle, before persisting incidents, query
`sage.action_log` for actions executed within `trust.rollback_window_minutes`
(default 30). Attach matching actions to every new or updated incident as
`RecentSageActions`. This is **purely informational** — no automated
rollback is triggered from this field alone.

### 13.2 Known Causal Path Decision Tree Branches

For specific, deterministic action→signal pairs, add decision tree branches
that identify pg_sage as a probable contributor. These are checked FIRST in
the relevant trees, before other branches.

```
Signal: log_out_of_memory
  -> NEW BRANCH (checked first):
     Check: sage.action_log contains action_type="work_mem_hint" OR
            action_type="work_mem_role_promotion" within rollback window?
       -> YES: ROOT CAUSE: "pg_sage raised work_mem to {value} for
                {role/query} {minutes} minutes ago. This may have
                contributed to memory exhaustion. Consider rolling back:
                {rollback_sql}"
                Severity: critical, Confidence: 0.85
                ActionRisk: "low" (rollback is a SET, not DDL)
                RollbackSQL: from action_log.rollback_sql
  -> (existing branches follow)

Signal: log_disk_full
  -> NEW BRANCH (checked first):
     Check: sage.action_log contains action_type="create_index" within
            rollback window?
       -> YES: ROOT CAUSE: "pg_sage created index {index_name}
                {minutes} minutes ago. If the index is large, it may
                have contributed to disk exhaustion. Review index size
                and consider: {rollback_sql}"
                Severity: critical, Confidence: 0.7
                Note: Lower confidence — disk full is often multi-cause
  -> (existing branches follow)

Signal: lock_contention
  -> NEW BRANCH (checked first):
     Check: sage.action_log contains action_type="vacuum_full" OR
            action_type="reindex" within rollback window AND the
            affected table matches a table in the lock chain?
       -> YES: ROOT CAUSE: "pg_sage's {action_type} on {table} is
                holding AccessExclusiveLock, blocking {n} sessions.
                The operation should complete in approximately
                {estimate}. If urgent, cancel via:
                SELECT pg_cancel_backend({pid});"
                Severity: critical, Confidence: 0.95
  -> (existing branches follow)

Signal: cache_hit_ratio_drop
  -> NEW BRANCH:
     Check: sage.action_log contains action_type="create_index" on a
            large table within rollback window?
       -> YES: ROOT CAUSE: "pg_sage created index on {table}
                {minutes} minutes ago. Large index builds can evict
                cached pages from shared_buffers, temporarily dropping
                cache hit ratio. This should recover within a few
                cycles as the working set is re-cached."
                Severity: warning, Confidence: 0.8
                Note: Transient — do NOT rollback the index for this
  -> (existing branches follow)

Signal: query regression (detected by existing metric signals)
  -> NEW BRANCH:
     Check: sage.action_log contains action_type="drop_index" within
            rollback window AND the dropped index was on a table
            referenced by the regressed query?
       -> YES: ROOT CAUSE: "pg_sage dropped index {index_name} on
                {table} {minutes} minutes ago. Query {queryid}
                subsequently regressed from {prev_mean}ms to
                {curr_mean}ms. The index may have been needed for
                infrequent but critical queries. Rollback:
                {rollback_sql}"
                Severity: critical, Confidence: 0.9
                ActionRisk: "low" (recreating index is safe)
  -> (existing branches follow)
```

### 13.3 Anti-Oscillation Guard

Risk: pg_sage detects problem → acts → RCA blames the action → rollback →
original problem resurfaces → pg_sage acts again → loop.

Mitigation: When RCA identifies a pg_sage action as a probable contributor
and the existing rollback mechanism fires:

1. The action family (e.g., "work_mem_hint for role X") enters
   `trust.rollback_cooldown_days` cooldown (existing mechanism).
2. The incident is annotated with `Source: "self_action"` so it can be
   distinguished from external incidents in dashboards and alerts.
3. If the same action family is rolled back twice within 30 days, emit a
   finding: "Recurring self-caused incident. The optimization and its
   rollback are oscillating. Manual review required." This prevents infinite
   loops.
