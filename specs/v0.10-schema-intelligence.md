## Thesis

v0.9 gave pg_sage the ability to detect and explain **runtime** problems (RCA from
metrics and logs). v0.10 gives it the ability to detect and prevent **structural**
problems — the schema-level mistakes that cause runtime incidents in the first place.

Every company that suffered a migration outage (Heroku, GoCardless, Doctolib, GitLab,
Basecamp, Duffel) built their own framework-specific safety tool afterward. pg_sage
consolidates all of these protections into a single database-level agent that works
regardless of application framework, ORM, or migration tool.

**Core differentiator**: Static linters (Squawk, Atlas, dryrun) analyze SQL files.
SaaS monitors (pganalyze, Datadog) observe metrics. App-level tools (Bullet, django-silk)
require instrumentation. pg_sage is the only tool that combines **runtime catalog access**
+ **live lock/activity awareness** + **LLM intelligence** + **trust-ramped execution**.

## 1. DDL Safety Advisor

### 1.1 Problem Statement

DDL-induced outages are the #1 self-inflicted PostgreSQL failure mode. A 4-hour
production lockout from a single CHECK constraint on 84M rows. A 15-second API
outage from a FK constraint locking a parent table. A 2-hour outage from anti-wraparound
vacuum colliding with partition DDL. These are documented, preventable incidents.

Existing tools operate at CI time (Squawk, Atlas) or framework time (strong_migrations).
None operate at the database level with real-time awareness of table sizes, lock state,
active queries, and replication lag.

### 1.2 Architecture

```
                          +-----------------+
                          |  DDL Intercept  |
                          |  (event trigger |
                          |   OR pg_audit   |
                          |   OR log parse) |
                          +--------+--------+
                                   |
                                   v
                          +-----------------+
                          |  DDL Classifier |
                          |  (lock level,   |
                          |   rewrite?,     |
                          |   safe alt?)    |
                          +--------+--------+
                                   |
                                   v
                          +-----------------+
                          |  Risk Assessor  |
                          |  (table size,   |
                          |   active locks, |
                          |   repl lag,     |
                          |   query load)   |
                          +--------+--------+
                                   |
                                   v
                          +-----------------+
                          |  Advisor Output |
                          |  (safe DDL,     |
                          |   risk score,   |
                          |   RCA incident) |
                          +-----------------+
```

### 1.3 DDL Detection

pg_sage detects DDL through **log-based detection** (extending the v0.9.1 logwatch
infrastructure), not event triggers. This avoids requiring superuser privileges or
modifying the target database's event trigger configuration.

**Detection method**: Parse `log_statement = 'ddl'` entries from PostgreSQL logs.
The logwatch parser already handles structured log lines; add a new signal category
for DDL statements.

**Prerequisites**: DDL log detection requires: (1) `log_statement` set to `'ddl'` or `'all'`, AND (2) the v0.9.1 logwatch infrastructure is active (which requires csvlog or jsonlog — jsonlog is PG15+, csvlog works on PG12+). If logwatch is disabled, DDL detection falls back to `activity_polling` mode only. If `log_statement` is `'none'` or `'mod'`, DDL statements other than DML are not logged; emit a startup warning: 'log_statement is not set to ddl or all; DDL log detection disabled, using activity polling only.'

**Log line patterns to match**:
- `statement: ALTER TABLE ...`
- `statement: CREATE INDEX ...`
- `statement: DROP ...`
- `statement: CREATE TABLE ... PARTITION OF ...`
- `statement: REFRESH MATERIALIZED VIEW ...`

**Limitation**: `log_statement = 'ddl'` does NOT log DDL executed inside PL/pgSQL functions or procedures — it logs the `CALL` statement only. DDL within functions is only caught by `activity_polling` if it's long-running.

**Fallback**: If `log_statement` is not set to `'ddl'` or `'all'`, the advisor
operates in **passive mode** — scanning `pg_stat_activity` on each analysis cycle
for active DDL commands. This catches long-running DDL but misses fast operations.

**Config**:
```yaml
migration:
  enabled: true
  mode: "advisory"           # advisory only in v0.10
  log_detection: true        # parse DDL from PG logs
  activity_polling: true     # scan pg_stat_activity for active DDL
  poll_interval_seconds: 5   # how often to poll during active DDL
```

### 1.4 DDL Classification Rules

Each detected DDL statement is classified by lock level, rewrite requirement, and
whether a safe alternative exists. Rules are deterministic — no LLM needed.

**SQL parsing strategy**: Use `pg_query_go` (Go bindings for libpg_query) for AST-based DDL classification. This is a **hard requirement**, not optional — regex/tokenization of PostgreSQL DDL is unreliable against multi-line strings, nested comments, standard-conforming strings (`E'...'`), and dollar-quoted bodies. `pg_query_go` provides a reliable parse tree, correctly handling all edge cases. Accept the CGo dependency. The AST is also used by N+1 detection (Section 3.2) for WHERE clause analysis.

| Rule ID | Pattern | Lock Level | Safe Alternative |
|---------|---------|------------|------------------|
| `ddl_index_not_concurrent` | `CREATE INDEX` without `CONCURRENTLY` on table > 10K rows | SHARE | `CREATE INDEX CONCURRENTLY` |
| `ddl_constraint_not_valid` | `ADD CONSTRAINT ... CHECK` without `NOT VALID` | ACCESS EXCLUSIVE (scan) | `NOT VALID` + `VALIDATE CONSTRAINT` |
| `ddl_fk_not_valid` | `ADD CONSTRAINT ... FOREIGN KEY` without `NOT VALID` | SHARE ROW EXCLUSIVE (scan) | `NOT VALID` + `VALIDATE CONSTRAINT` |
| `ddl_set_not_null` | `ALTER COLUMN ... SET NOT NULL` without existing valid CHECK (col IS NOT NULL) constraint | ACCESS EXCLUSIVE (scan) | Add CHECK (col IS NOT NULL) NOT VALID, VALIDATE, then SET NOT NULL (metadata-only on PG12+) |
| `ddl_alter_type_rewrite` | `ALTER COLUMN ... TYPE` requiring table rewrite | ACCESS EXCLUSIVE (rewrite) | New column + trigger + backfill + swap |
| `ddl_add_column_volatile_default` | `ADD COLUMN ... DEFAULT <volatile>` | ACCESS EXCLUSIVE (rewrite) | Add nullable, backfill in batches, add constraint |
| `ddl_drop_column` | `DROP COLUMN` | ACCESS EXCLUSIVE | Advisory: ensure no dependent views/functions |
| `ddl_drop_table` | `DROP TABLE` | ACCESS EXCLUSIVE | Advisory: verify no references |
| `ddl_reindex_not_concurrent` | `REINDEX` without `CONCURRENTLY` (PG12+) | ACCESS EXCLUSIVE | `REINDEX CONCURRENTLY` |
| `ddl_vacuum_full` | `VACUUM FULL` | ACCESS EXCLUSIVE | `pg_repack` or regular VACUUM |
| `ddl_refresh_not_concurrent` | `REFRESH MATERIALIZED VIEW` without `CONCURRENTLY` | ACCESS EXCLUSIVE | `REFRESH ... CONCURRENTLY` (requires unique index) |
| `ddl_cluster` | `CLUSTER` | ACCESS EXCLUSIVE | Advisory: schedule during maintenance window |
| `ddl_set_tablespace` | `ALTER TABLE ... SET TABLESPACE` | ACCESS EXCLUSIVE (file move) | Advisory: schedule during low traffic |
| `ddl_add_column_not_null` | `ADD COLUMN ... NOT NULL` without DEFAULT (PG < 11: rewrite; PG11+: metadata-only) | ACCESS EXCLUSIVE (rewrite on PG < 11) | PG11+: safe. PG < 11: add nullable, backfill, SET NOT NULL |
| `ddl_attach_partition_no_check` | `ATTACH PARTITION` without matching CHECK constraint on child | ACCESS EXCLUSIVE (child) + SHARE UPDATE EXCLUSIVE (parent) | Add CHECK constraint matching partition bounds on child BEFORE attach. Without it, PG table-scans the child under lock. |
| `ddl_missing_lock_timeout` | Any ACCESS EXCLUSIVE DDL without preceding `SET lock_timeout` | — | Recommend `SET lock_timeout = '5s'` |

**Volatile default classification** (`ddl_add_column_volatile_default`): A default expression is considered volatile if it contains: (1) any function call NOT in the known-safe allowlist (`now()`, `current_timestamp`, `clock_timestamp()`, `gen_random_uuid()`, `uuid_generate_v4()`), (2) any subquery, or (3) `nextval()` calls. Alternatively, the classifier may query `pg_proc.provolatile` for function references. Note: PG11+ treats immutable/stable defaults as metadata-only operations; pg_sage relies on this PG11+ behavior.

**Table size thresholds**: Rules that reference "table > 10K rows" use
`pg_class.reltuples` for fast estimation. The threshold is configurable:
```yaml
migration:
  ddl_row_threshold: 10000     # rows below which DDL is considered low-risk
```

**Version-gated rules**: Each DDL rule's safe alternative is gated on the target PostgreSQL version. `ddl_reindex_not_concurrent` recommends `REINDEX CONCURRENTLY` only on PG12+; on PG < 12, recommend scheduling during maintenance window. `ddl_add_column_volatile_default` applies PG11+ behavior (immutable/stable defaults are metadata-only); on PG < 11 ALL defaults cause a rewrite. The advisor output never recommends features unavailable on the target database. A `min_pg_version` field is included in each rule definition.

### 1.5 Risk Assessment

When a DDL statement is classified as potentially dangerous, the Risk Assessor
gathers runtime context:

```go
type DDLRisk struct {
    Statement       string
    RuleID          string
    LockLevel       string   // "ACCESS EXCLUSIVE", "SHARE ROW EXCLUSIVE", etc.
    RequiresRewrite bool
    TableName       string
    TableSizeBytes  int64
    EstimatedRows   int64
    ActiveQueries   int      // queries currently touching this table
    LongestQuerySec float64  // longest running query on this table
    PendingLocks    int      // lock requests queued on this table
    ReplicationLag  float64  // seconds behind primary
    RiskScore       float64  // 0.0-1.0 composite score
    SafeAlternative string   // SQL for safe approach
    EstimatedLockMs int64    // estimated lock hold time
}
```

**Replication lag detection**: Determine role via `pg_is_in_recovery()`. On primary: query `pg_stat_replication` for `replay_lag` of slowest standby. On standby: compute lag from `pg_last_wal_receive_lsn() - pg_last_wal_replay_lsn()`. Requires `pg_monitor` role or equivalent permissions. Skip `repl_factor` for UNLOGGED tables (`pg_class.relpersistence = 'u'`) since they don't generate WAL.

**Lock queue cascade detection**: Before computing the risk score, check for long-running transactions holding ANY lock on the target table. If an ACCESS EXCLUSIVE DDL must wait for an existing lock, ALL subsequent queries queue behind it — exhausting connection pools in seconds. If any transaction on the target table has been running > 5 seconds, immediately set `lock_queue_factor = 1.0` and add a causal chain entry: "Existing long-running transaction will cause lock queue cascade."

**Transactional DDL awareness**: PostgreSQL DDL is transactional — the ACCESS EXCLUSIVE lock is held until `COMMIT`. If log detection sees DDL inside a transaction (`BEGIN; ALTER TABLE ...; UPDATE ...; COMMIT;`), warn that the lock is held for the full transaction duration, not just the DDL execution.

**Risk score computation**:
```
base_risk = lock_level_weight * rewrite_weight
table_factor = log10(estimated_rows) / 10       # 0.0 for 1 row, 1.0 for 10B rows
activity_factor = min(active_queries / 100, 1.0)
repl_factor = min(replication_lag / 30, 1.0)
lock_queue_factor = min(pending_locks / 10, 1.0)

risk_score = base_risk * max(0.1,
    0.4 * table_factor
  + 0.3 * activity_factor
  + 0.2 * lock_queue_factor
  + 0.1 * repl_factor)
```

Where `lock_level_weight`:
- ACCESS EXCLUSIVE: 1.0
- SHARE ROW EXCLUSIVE: 0.7
- SHARE: 0.5
- SHARE UPDATE EXCLUSIVE: 0.3

And `rewrite_weight`:
- Requires rewrite: 1.0
- Scan only (no rewrite): 0.6
- Metadata only: 0.2

The `max(0.1, ...)` floor ensures ACCESS EXCLUSIVE DDL always produces a non-zero risk score, even on idle small tables. Without this floor, metadata-only operations on 1-row tables with no activity produce risk_score = 0.0, which would suppress advisory output for ACCESS EXCLUSIVE operations.

### 1.6 Advisor Output

For risk_score > 0.3, the advisor produces an **RCA incident** (source:
`"schema_advisor"`):

```go
Incident{
    Source:         "schema_advisor",
    Severity:       severityFromRisk(risk_score), // >0.7 critical, >0.3 warning
    RootCause:      "DDL statement requires ACCESS EXCLUSIVE lock on orders (84M rows)",
    Confidence:     0.95,  // deterministic rules
    CausalChain:    []CausalLink{
        {Order: 1, Description: "ALTER TABLE orders ADD CONSTRAINT ... acquires ACCESS EXCLUSIVE lock"},
        {Order: 2, Description: "84M rows require full table scan for constraint validation (~4 min)"},
        {Order: 3, Description: "12 active queries will be blocked, lock queue will cascade"},
    },
    RecommendedSQL: "ALTER TABLE orders ADD CONSTRAINT ... NOT VALID;\nALTER TABLE orders VALIDATE CONSTRAINT ...;",
    ActionRisk:     "safe",
    SignalIDs:      []string{"ddl_constraint_not_valid"},
}
```

### 1.7 LLM Enhancement (Tier 2)

For DDL statements that don't match any deterministic rule but appear risky (e.g.,
complex multi-statement migrations, custom PL/pgSQL DDL), the LLM is consulted:

**Prompt template**:
```
You are a PostgreSQL migration safety expert. Analyze this DDL statement for safety
risks on PostgreSQL {{.Version}}.

Statement: {{.SQL}}
Table: {{.TableName}} ({{.EstimatedRows}} rows, {{.TableSizeBytes}} bytes)
Active queries on table: {{.ActiveQueries}}
Current lock state: {{.LockState}}

Assess:
1. Lock level acquired
2. Whether a table rewrite is required
3. Estimated duration
4. Safe alternative if the operation is risky
5. Risk score (0.0-1.0)

Respond in JSON...
```

The LLM path uses the same token budget and rate limiting as the existing Tier 2
RCA system. **Pipeline integration**: Schema intelligence LLM calls share the rate limiter with Tier 2 RCA calls. RCA Tier 2 has priority; schema advisory calls are queued behind active RCA analysis. Each prompt template defines its JSON response schema explicitly.

### 1.8 Active Mode (Future — Trust Level: auto)

When trust_level is `auto` and migration mode is `active`, pg_sage can:
1. Detect a dangerous DDL about to execute (via event trigger, requires extension)
2. Cancel the original statement (`pg_cancel_backend`)
3. Execute the safe alternative automatically
4. Log the transformation for audit

This is a future capability gated on the trust ramp. v0.10 ships with advisory
mode only. If `mode: "active"` is set in config, log a warning and fall back to `advisory`.

## 2. Schema Anti-Pattern Detection

### 2.1 Problem Statement

Schema anti-patterns are silent performance killers. A missing FK index causes a
3,000x performance cliff on DELETE operations. Wrong timestamp types cause timezone
bugs. Integer sequences approaching overflow cause 5-hour outages (Basecamp).
JSONB columns in JOINs cause 2,000x slowdowns (Heap).

No existing tool continuously monitors for these patterns with runtime impact
correlation. Static linters (SchemaCrawler, Squawk) flag patterns without knowing
table sizes or query frequencies. pg_sage correlates schema structure with
`pg_stat_statements` usage and `pg_stat_user_tables` metrics to prioritize by
actual impact.

### 2.2 Architecture

The schema linter runs as a **periodic scan** within the existing analysis loop.
It does NOT run every cycle — schema changes are infrequent, so scanning every
`schema_scan_interval` (default: 1 hour) is sufficient.

```go
// internal/schema/lint/linter.go
type Linter struct {
    pool     *pgxpool.Pool
    rules    []Rule
    cfg      *config.SchemaLintConfig
    logger   func(level, msg string, args ...any)
}

type Rule interface {
    ID()          string
    Name()        string
    Severity()    string           // "info", "warning", "critical"
    Category()    string           // "safety", "performance", "correctness", "convention"
    Check(ctx context.Context, pool *pgxpool.Pool) ([]Finding, error)
}

type Finding struct {
    RuleID      string
    Schema      string
    Table       string
    Column      string             // empty if table-level
    Index       string             // empty if not index-related
    Severity    string
    Category    string
    Description string
    Impact      string             // estimated impact description
    TableSize   int64              // bytes, for prioritization
    QueryCount  int64              // from pg_stat_statements, if relevant
    Suggestion  string             // human-readable fix
    SQL         string             // executable fix DDL
}
```

**Config**:
```yaml
schema_lint:
  enabled: true
  scan_interval_minutes: 60
  rules:
    disabled: []                    # rule IDs to skip, e.g. ["lint_serial_usage"]
    min_table_rows: 1000           # ignore findings on tables below this (noise suppression)
  severity_override: {}            # rule_id -> severity
```

### 2.2.1 Finding Type Integration

Schema lint uses its own `lint.Finding` type internally (for persistence to `sage.schema_findings`), but surfaces results as `analyzer.Finding` objects when feeding into the incident pipeline. A `ToAnalyzerFinding()` method on `lint.Finding` handles the conversion. This keeps the lint package self-contained while integrating with the existing incident flow.

#### 2.3.1 Safety Rules (prevent incidents)

```sql
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    rule_id       TEXT NOT NULL,
    schema_name   TEXT NOT NULL,
    table_name    TEXT NOT NULL,
    column_name   TEXT,           -- NULL if table-level
    index_name    TEXT,           -- NULL if not index-related
    severity      TEXT NOT NULL CHECK (severity IN ('info', 'warning', 'critical')),
    category      TEXT NOT NULL CHECK (category IN ('safety', 'performance', 'correctness', 'convention')),
    description   TEXT NOT NULL,
    suggestion    TEXT,
    suggested_sql TEXT,
    table_size    BIGINT,
    impact_score  REAL,           -- 0.0-1.0 computed impact
    first_seen    TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_seen     TIMESTAMPTZ NOT NULL DEFAULT now(),
    resolved_at   TIMESTAMPTZ,    -- set when finding no longer detected
    suppressed    BOOLEAN NOT NULL DEFAULT false,
    UNIQUE (rule_id, schema_name, table_name, COALESCE(column_name, ''), COALESCE(index_name, ''))
);

CREATE INDEX idx_findings_severity ON sage.schema_findings (severity) WHERE resolved_at IS NULL;
CREATE INDEX idx_findings_table ON sage.schema_findings (schema_name, table_name);
```

On each scan:
- New findings are inserted
- Existing findings have `last_seen` updated
- Findings no longer detected have `resolved_at` set
- Suppressed findings (user dismissed) are skipped in incident generation

**Upsert behavior**: Uses `INSERT ... ON CONFLICT DO UPDATE` on the UNIQUE key. On conflict, updates `last_seen`, `description`, `suggestion`, `suggested_sql`, `table_size`, and `impact_score`. The `first_seen` timestamp is preserved.

**Autovacuum cross-reference** (`lint_bloated_table`): The bloat rule queries `pg_stat_user_tables.last_autovacuum` and per-table autovacuum settings (`pg_class.reloptions`). If autovacuum ran recently and bloat persists, the suggestion is 'consider pg_repack or manual VACUUM FULL during maintenance' rather than 'tune autovacuum.'

## 3. N+1 Query Detection

### 3.1 Problem Statement

N+1 queries are the #1 ORM performance problem. Every existing detection tool
operates at the application layer (Bullet for Rails, nplusone for Python, APM tools).
**No tool detects N+1 at the database level.** This is a greenfield opportunity.

pg_sage has `pg_stat_statements` access, which shows normalized query patterns with
call counts. A query called 10,000 times with `rows = 1` per call is almost certainly
an N+1 pattern. We can detect this without any application instrumentation, across
all applications hitting the database, regardless of language or framework.

### 3.2 Detection Algorithm

**Step 1: Identify N+1 candidates from pg_stat_statements**

```sql
-- Candidate: high call count, low rows per call, single-table lookup pattern
SELECT
    queryid,
    query,
    calls,
    mean_exec_time,
    rows,
    CASE WHEN calls > 0 THEN rows::float / calls ELSE 0 END AS rows_per_call,
    calls * mean_exec_time AS total_time_ms
FROM pg_stat_statements
WHERE calls > :min_calls                -- default: 100
  AND mean_exec_time < :max_mean_ms     -- default: 5.0 (fast individual queries)
  AND (rows::float / NULLIF(calls, 0)) < :max_rows_per_call  -- default: 5.0
  AND query ~* 'SELECT.*FROM.*WHERE.*=\s*\$'  -- parameterized single-row lookup (see note below)
ORDER BY calls * mean_exec_time DESC
LIMIT 50;
```

**Query pattern matching**: The regex `query ~* 'SELECT.*FROM.*WHERE.*=\s*\$'` is a first-pass filter only — it misses multi-condition queries like `WHERE id = $1 AND tenant_id = $2`. Preferred approach: if `pg_query_go` is available (see Section 1.4), parse the query AST and check for WHERE clauses with equality conditions on parameters. The regex serves as a fallback for environments where AST parsing is not viable.

**PG14+ optimization**: When PostgreSQL >= 14, add `AND toplevel` to the WHERE clause to exclude calls from within functions, triggers, and procedures. On PG12-13, the `toplevel` column is unavailable; accept degraded accuracy from inflated call counts.

**Query text truncation**: `pg_stat_statements` truncates query text at `track_activity_query_size` (default: 1024 bytes). For truncated queries where the `WHERE ... = $` pattern cannot be matched, multiply N+1 confidence by 0.5. Recommend setting `track_activity_query_size = 4096` for best results. At startup, if `track_activity_query_size < 2048`, log a warning.

**Step 2: Correlate with parent queries**

For each N+1 candidate, look for a "parent" query that:
- Touches the same table referenced by the candidate's FK
- Has a correlated call count: `ratio = child_calls / parent_calls` where `ratio > 2.0` and `(ratio - floor(ratio)) / ratio < ratio_tolerance` (default: 0.1, configurable as `n_plus_one.ratio_tolerance`)
- Both queries appear in the same time window

```go
type NplusOneCandidate struct {
    QueryID       int64
    Query         string           // normalized query text
    Calls         int64
    MeanExecMs    float64
    RowsPerCall   float64
    TotalTimeMs   float64
    ParentQueryID *int64           // suspected parent query, if identified
    ParentQuery   *string
    Ratio         float64          // candidate.calls / parent.calls
    EstSavingsMs  float64          // estimated time saved by batching
    Confidence    float64          // 0.0-1.0
}
```

**Step 3: Compute savings estimate**

```
batch_time_estimate = parent_calls * single_batch_exec_time
current_total_time = candidate_calls * candidate_mean_exec_time
estimated_savings = current_total_time - batch_time_estimate
```

### 3.3 Confidence Scoring

| Signal | Weight | Description |
|--------|--------|-------------|
| High calls, low rows_per_call | 0.3 | Core N+1 pattern |
| Parameterized single-row WHERE | 0.2 | `WHERE id = $1` pattern |
| Parent query correlation found | 0.25 | Parent returns N rows, child called N times |
| Table has FK relationship | 0.15 | FK between parent and child tables |
| Consistent across cycles | 0.1 | Pattern persists across multiple analysis cycles |

Confidence >= 0.6: report as N+1 candidate
Confidence >= 0.8: report as high-confidence N+1

### 3.4 LLM Enhancement

For high-confidence N+1 detections, the LLM generates:
1. **Explanation**: Why this is an N+1 and what's happening
2. **Batched alternative**: Rewrite using `WHERE id = ANY($1)` or JOIN
3. **ORM hint**: If the query pattern matches a known ORM (Django, Rails, SQLAlchemy),
   suggest the ORM-specific fix (`.select_related()`, `.includes()`, etc.)

**Prompt template**:
```
These two PostgreSQL queries appear to form an N+1 pattern:

Parent query ({{.ParentCalls}} calls):
{{.ParentQuery}}

Child query ({{.ChildCalls}} calls, {{.ChildRowsPerCall}} rows/call):
{{.ChildQuery}}

The child query is called ~{{.Ratio}}x per parent call.
Total wasted time: {{.TotalTimeMs}}ms per cycle.

1. Confirm this is an N+1 pattern and explain why.
2. Write a single batched query that replaces the child loop.
3. If this looks like an ORM pattern, suggest the ORM fix.

Respond in JSON...
```


### 3.6 Output

N+1 detections surface as:
1. **REST API**: `GET /api/n-plus-one` — list of detected patterns with savings
2. **RCA incidents**: High-confidence patterns with significant time waste (>1s total)
   produce incidents (source: `"n_plus_one"`)
3. **Dashboard**: N+1 tab showing patterns ranked by total wasted time

### 3.7 False Positive Mitigation

Not all high-call-count queries are N+1:
- **Cron/batch jobs**: Legitimate high-frequency queries from batch processors.
  Mitigation: require consistent call patterns across multiple cycles, not just
  one spike.
- **Connection pool health checks**: `SELECT 1` patterns. Mitigation: exclude
  queries with no FROM clause.
- **Metrics collectors**: Monitoring queries. Mitigation: allow user to mark
  queries as "known OK" via API (`POST /api/n-plus-one/{id}/suppress`).
- **Pagination**: Sequential queries with OFFSET. Mitigation: detect OFFSET/LIMIT
  pattern and exclude.
- **Connection poolers (PgBouncer, Odyssey)**: May cause false positive correlations when unrelated applications share the same database. Consider the `min_confidence` threshold and manual suppression for environments with heavy multiplexing. On PG14+, filtering `toplevel = true` reduces noise.

**Known limitation — false negatives**: `pg_stat_statements` aggregates queries globally. If `SELECT * FROM users WHERE id = $1` is called 10,000 times by an N+1 loop AND 50,000 times by a healthy API endpoint, the N+1 pattern is obfuscated in the aggregate metrics. pg_sage will catch pure, unmixed N+1 patterns but has a high false-negative rate for heavily reused singleton queries. This is an inherent limitation of database-level detection without per-session tracking.

### 3.8 Persistence

```sql
CREATE TABLE IF NOT EXISTS sage.n_plus_one_detections (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    parent_hash   BIGINT,
    parent_text   TEXT,
    calls         BIGINT,
    mean_exec_ms  REAL,
    rows_per_call REAL,
    confidence    REAL,
    est_savings_ms REAL,
    first_seen    TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_seen     TIMESTAMPTZ NOT NULL DEFAULT now(),

```

**Suppression identity**: N+1 suppressions use `query_hash` (a hash of the normalized query text) rather than `pg_stat_statements.queryid`, which changes across stats resets and PG upgrades. The suppression survives stats resets.

## 4. Materialized View Intelligence

### 4.1 Problem Statement

PostgreSQL materialized views have no built-in auto-refresh, staleness detection,
or lifecycle management. Teams resort to pg_cron (not available on all managed
services), application triggers (causes cascading disasters — documented incident:
10-minute insert latency), or manual refresh. REFRESH without CONCURRENTLY blocks
all reads. CONCURRENTLY requires a unique index that teams often forget.

No existing tool provides matview lifecycle management. pg_ivm (incremental) has
severe limitations and isn't available on managed services. This is greenfield.

### 4.2 Discovery and Inventory

On each schema scan, discover all materialized views:

```sql
SELECT
    schemaname,
    matviewname,
    matviewowner,
    ispopulated,
    definition,
    pg_total_relation_size((schemaname || '.' || matviewname)::regclass) AS total_size,
    -- Check for unique index (CONCURRENTLY readiness)
    EXISTS (
        SELECT 1 FROM pg_index i
        JOIN pg_class c ON i.indexrelid = c.oid
        WHERE i.indrelid = (schemaname || '.' || matviewname)::regclass
          AND i.indisunique
    ) AS has_unique_index,
    -- Last refresh time (from pg_stat_user_tables if populated)
    s.last_analyze AS last_stats_update
FROM pg_matviews m
LEFT JOIN pg_stat_user_tables s
  ON s.schemaname = m.schemaname AND s.relname = m.matviewname
WHERE schemaname NOT IN ('pg_catalog', 'information_schema');
```

### 4.3 Staleness Detection

Track base table modification rates and compare to matview refresh times.

**Base table resolution**: Resolve base tables via recursive CTE on `pg_depend` (where `deptype = 'n'` linking the matview's rewrite rule to dependent relations). Do NOT parse SQL definition strings — aliases, CTEs, and schema paths make text parsing unreliable. This also catches nested dependencies (matview -> view -> table).

```go
type MatviewHealth struct {
    Schema          string
    Name            string
    Definition      string
    SizeBytes       int64
    HasUniqueIndex  bool          // can use CONCURRENTLY
    IsPopulated     bool
    LastRefresh     *time.Time    // tracked by pg_sage
    BaseTables      []string      // resolved via pg_depend recursive CTE (NOT parsed from SQL)
    BaseTableMods   int64         // sum of n_tup_ins + n_tup_upd + n_tup_del since last refresh
    StalenessScore  float64       // 0.0 (fresh) to 1.0 (very stale)
    QueryCount      int64         // references in pg_stat_statements
    RefreshCostMs   int64         // avg refresh duration (tracked by pg_sage)
    CostBenefit     float64       // (query_speedup * query_count) / refresh_cost
}
```

**Refresh tracking**: PostgreSQL does not expose matview last refresh time in any catalog. pg_sage tracks refreshes in `sage.matview_refresh_log`:
```sql
CREATE TABLE IF NOT EXISTS sage.matview_refresh_log (
    matview_schema TEXT NOT NULL,
    matview_name   TEXT NOT NULL,
    refreshed_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    duration_ms    BIGINT,
    concurrent     BOOLEAN,
    PRIMARY KEY (matview_schema, matview_name)
);
```
This table is updated: (a) after any pg_sage-initiated refresh, and (b) when a `REFRESH MATERIALIZED VIEW` statement is detected in logs. For matviews refreshed externally without logging, staleness cannot be tracked — `LastRefresh` remains NULL. On first discovery, `LastRefresh` is NULL (unknown).

**Staleness score**:
```
time_factor = hours_since_refresh / configured_max_staleness_hours
mod_factor = base_table_mods / (base_table_rows * 0.1)  # >10% change = fully stale
staleness_score = max(time_factor, mod_factor)            # clamped to 1.0
```

### 4.4 Refresh Management

When staleness_score exceeds the configured threshold (default: 0.7) and trust_level
permits:

1. Check disk space: CONCURRENTLY requires ~2x matview size in temp space
2. Verify unique index exists (for CONCURRENTLY)
3. Check for active queries that would conflict
4. Execute `REFRESH MATERIALIZED VIEW CONCURRENTLY <name>`
5. Track refresh duration for future cost estimation
6. If CONCURRENTLY fails (no unique index), fall back to advisory: recommend
   creating a unique index

**Config**:
```yaml
matview:
  enabled: true
  staleness_threshold: 0.7       # trigger refresh at this staleness score
  max_refresh_duration_minutes: 30  # abort refresh if exceeds this
  auto_refresh: false             # requires trust_level: auto
  scan_interval_minutes: 15
```

### 4.5 Matview Recommendations (LLM-Enhanced)

Analyze `pg_stat_statements` for patterns that suggest a matview would help:
- Repeated complex aggregations (`GROUP BY` with `SUM/COUNT/AVG`) on the same tables
- Slow queries (>500ms mean) with stable result sets
- Queries executing > 100x/day with identical structure

The LLM evaluates whether a matview is the right solution vs. a covering index
or query rewrite:

**Prompt template**:
```
This PostgreSQL query executes {{.Calls}} times per day with {{.MeanExecMs}}ms
mean execution time:

{{.Query}}

Table sizes: {{.TableSizes}}

Should this be a materialized view? Consider:
1. Would a covering index solve this more efficiently?
2. How often does the underlying data change? ({{.BaseTableModRate}} mods/hour)
3. What is the refresh cost vs. per-query savings?

Respond with recommendation: "matview", "covering_index", "query_rewrite", or
"no_change", with rationale and DDL.
```

### 4.6 Anti-Pattern Detection

Detect matview anti-patterns:
- **Trigger-based refresh**: Detect triggers on base tables that call `REFRESH
  MATERIALIZED VIEW`. Alert as critical — this caused a documented 10-minute
  insert latency incident.
- **Missing unique index**: Flag matviews without unique indexes that therefore
  cannot use CONCURRENTLY.
- **Unused matviews**: Matviews with zero references in `pg_stat_statements` for
  >7 days. Recommend dropping.
- **Oversized matviews**: Matviews larger than the base tables they aggregate.
- **Matview dependency chains**: Detect matviews that depend on other matviews via `pg_depend` and `pg_rewrite`. When computing staleness for a dependent matview, propagate staleness: if `mv_detail` is stale, `mv_summary` is at least as stale. When auto-refreshing, refresh dependencies first (topological sort). Circular dependencies are flagged as an anti-pattern.

## 5. Sequence Overflow Monitor

### 5.1 Problem Statement

Integer sequence overflow is a ticking time bomb. Basecamp suffered a 5-hour
read-only outage. Buildkite proactively migrated a 2B+ row table. Crunchy Data
reports helping multiple clients navigate overflow emergencies. The detection is
trivial — a single SQL query — but no tool continuously monitors this.

### 5.2 Detection

Run every schema lint cycle:

```sql
SELECT
    seqrelid::regclass AS sequence_name,
    seqtypid::regtype AS data_type,
    CASE seqtypid::regtype::text
        WHEN 'smallint' THEN 32767
        WHEN 'integer'  THEN 2147483647
        WHEN 'bigint'   THEN 9223372036854775807
    END AS max_value,
    last_value,
    ROUND(
        100.0 * last_value /
        CASE seqtypid::regtype::text
            WHEN 'smallint' THEN 32767
            WHEN 'integer'  THEN 2147483647
            WHEN 'bigint'   THEN 9223372036854775807
        END, 2
    ) AS pct_used
FROM pg_sequences
WHERE last_value IS NOT NULL
ORDER BY pct_used DESC;
```

**Permissions**: `pg_sequences.last_value` requires the pg_sage database role to have `pg_read_all_sequences` privilege (PG15+: included in `pg_monitor`) or ownership of the sequences. On managed services, ensure the role has `pg_monitor` or `pg_read_all_sequences`. If `last_value` is NULL for all sequences, log a warning about insufficient privileges.

### 5.3 Time-to-Exhaustion Projection

Track insertion rate over time to project when the sequence will exhaust:

```go
func (m *SequenceMonitor) TimeToExhaustion(seqName string) time.Duration {
    // Delta of last_value over past N cycles
    // Project linearly to max_value
    // Return estimated time remaining
}
```

**Package location**: The `SequenceMonitor` resides in `internal/schema/lint/` as an extension of the lint rule, not a separate monitor. The existing `internal/analyzer/rules_sequence.go` handles per-cycle detection and remains as-is. The lint package adds time-to-exhaustion projection and the expanded threshold tiers.

### 5.4 Alert Thresholds

| Threshold | Severity | Action |
|-----------|----------|--------|
| > 50% used | info | Log finding, include in schema report |
| > 75% used | warning | Generate RCA incident |
| > 85% used | critical | Generate RCA incident with recommended migration DDL |
| > 95% used | critical | Urgent alert, include time-to-exhaustion estimate |

For critical findings, generate the safe migration DDL using the expand-contract
pattern (new bigint column + trigger + backfill + swap) via LLM.

## 6. Operational Constraints

### 6.1 pg_stat_statements Dependency

N+1 detection (Section 3), impact correlation (Section 2.4), and `lint_jsonb_in_joins` all require `pg_stat_statements`. At startup, check `shared_preload_libraries` for the extension. If absent, disable these features with a warning: "pg_stat_statements not loaded; N+1 detection and schema lint impact correlation disabled."

**Feature dependency matrix**:
| Feature | Requires pg_stat_statements | Catalog-only fallback |
|---------|---------------------------|----------------------|
| Schema lint (most rules) | No | Full functionality |
| lint_jsonb_in_joins | Yes | Disabled |
| Impact correlation | Yes | Disabled (findings still generated, no impact score) |
| N+1 detection | Yes | Entirely disabled |
| DDL advisor | No | Full functionality |
| Matview staleness | No | Full functionality |
| Sequence monitor | No | Full functionality |

### 6.2 Standby/Replica Behavior

Detect standby status via `pg_is_in_recovery()`. On standby databases:
- **Schema lint**: Runs normally (read-only catalog queries)
- **DDL advisor**: Disabled (no DDL on standbys)
- **N+1 detection**: Runs but findings include note: "Detected on standby; call counts reflect read-replica traffic only"
- **Matview refresh**: Disabled (read-only); staleness detection still runs but cannot remediate
- **Sequence monitor**: Runs normally (read-only queries)

### 6.3 Partitioned Tables

All features must be partition-aware:
- **DDL advisor**: Compute risk using `pg_total_relation_size()` which includes partitions. For `reltuples`, sum across `pg_class` for all `pg_inherits` children. A 0-row parent with million-row children must be assessed as a large table. **Critical**: Safe alternative generation must be partition-aware — `VALIDATE CONSTRAINT` on a partitioned parent does NOT validate children; the generated DDL must iterate over each child partition individually.
- **Schema lint**: Check children independently for missing indexes. Flag if a FK index exists on the parent but not on a child partition. FKs on partitioned tables are inherited by children in PG12+, but indexes are NOT automatically created on children.
- **Sequence monitor**: Already handles this (sequences are standalone objects).
- **N+1 detection**: Use partition-aware row counts — sum `reltuples` across parent and children.

### 6.4 Temporary Table Exclusion

Exclude schemas matching `pg_temp_%` and `pg_toast_temp_%` from all lint rules, DDL classification, and N+1 detection. Temporary tables are noise for all schema intelligence features.

### 6.5 Schema Filtering

All lint rules must be schema-aware. The `ForeignKey` collector type needs `SchemaName` and `ReferencedSchema` fields (pre-existing gap exposed by v0.10). Configuration:
```yaml
schema_lint:
  include_schemas: []  # empty = all non-system schemas
  exclude_schemas: ["pg_catalog", "information_schema", "pg_toast"]
```

### 6.6 Managed Service Compatibility

Managed PostgreSQL services (RDS, Aurora, Cloud SQL, AlloyDB) restrict several operations:
- **Log-based DDL detection**: Not available (no log file access). Falls back to activity polling automatically.
- **Extension availability**: `pg_stat_statements` may require manual enablement.
- **Recommendations**: Some recommendations (e.g., `pg_repack`, `ALTER SYSTEM`) are unavailable on managed services.

Configuration:
```yaml
migration:
  managed_service: "none"  # none | rds | aurora | cloudsql | alloydb
```
When set, filter out recommendations referencing unavailable features. Auto-detection: query `SHOW rds.extensions` (RDS/Aurora) or check for `cloudsql.enable_pgaudit` (Cloud SQL).

### 6.7 Concurrent DDL (TOCTOU)

Risk assessments reflect a point-in-time snapshot. Between assessment and advisory delivery, lock state may change. For multiple DDL statements detected in the same cycle, assess independently but flag as a group: "N ACCESS EXCLUSIVE DDL operations detected in the same window. Risk of cascading lock contention is elevated."

### 6.8 Minimum PostgreSQL Version

pg_sage v0.10 targets PostgreSQL 12+ as the minimum supported version. Version-specific behavior:
- PG12-13: No `toplevel` filter on pg_stat_statements (degraded N+1 accuracy)
- PG12-14: csvlog only for DDL log detection
- PG15+: jsonlog support, `pg_read_all_sequences` included in `pg_monitor`
- PG11: NOT officially supported; some features may work but are not tested

Each DDL rule includes a `min_pg_version` field. The advisor never recommends features unavailable on the target database version.

## Appendix A: Competitive Moat

This feature set creates three defensible moats:

1. **Runtime-aware schema linting**: Only pg_sage correlates schema anti-patterns
   with actual query frequency and table sizes. A missing FK index on a 10-row
   lookup table is noise; on a 10M-row table queried 1000x/sec, it's critical.
   Static linters cannot make this distinction.

2. **Database-level N+1 detection**: No existing tool detects N+1 from the database
   side. This works across ALL applications, ALL languages, ALL ORMs — no
   instrumentation required. The detection is framework-agnostic by design.

3. **Matview lifecycle management**: Complete greenfield. No tool provides discovery,
   staleness tracking, cost-benefit analysis, and safe refresh execution for
   materialized views.
