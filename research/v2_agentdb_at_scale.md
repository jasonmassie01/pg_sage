# AgentDB at Scale: The Autonomous DBA Layer for Agent-Deployed Databases

**Date:** 2026-06-10
**Scope:** What pg_sage's AgentDB subsystem should become once it serves databases deployed *by* agents at scale — ephemeral, hundreds-to-thousands spawned per hour, often poorly tuned, sporadically used, multi-tenant.
**Relationship to prior work:** [research/v1_agent_created_databases.md](./v1_agent_created_databases.md) covered the *workload* inside an agent DB (schema sprawl, memory hygiene, DDL gating, poisoning). This document covers the *control plane around* thousands of agent DBs — provisioning, tuning, lifecycle, isolation, and cost. The two compose: v1 is "what to do inside one agent DB," v2 is "how to operate ten thousand of them." Do not re-derive v1's findings here.

---

## 0. The gap, restated against the as-built system

pg_sage already *provisions* agent-requested databases on AWS RDS, GCP Cloud SQL, and Databricks Lakebase, with a real state machine, leases, TTL tags, cost/budget gates, and a verified-backup cleanup interlock ([docs/reverse_spec/05-agentdb.md](../docs/reverse_spec/05-agentdb.md)). What it does *not* do is operate those databases after birth: they are not in the fleet (no collector/analyzer), the reconciler that would sweep abandoned and in-flight deployments exists and is tested but **nothing schedules it**, chargeback is agent-self-reported with no metering, "archival" is a status not a data movement, and tenant rows are *tagged* but not *isolated* on reads. AgentDB is a provisioning ledger, not an autonomous DBA.

The thesis of this document: the demographic shift is already here — as of late 2025 over 80% of databases on Neon were created by AI agents, and 97% of branches ([SaaStr/Databricks](https://www.saastr.com/databricks-only-19-of-organizations-have-deployed-ai-agents-but-theyre-already-creating-97-of-databases/), cited in v1). The median agent DB is ephemeral, never tuned by a human, and lives in a fleet of thousands. The architectural template for serving it well is **Neon** — storage/compute separation that yields sub-second provisioning, ~300 ms resume from idle, and ~1 s copy-on-write branches ([Neon architecture](https://neon.com/docs/introduction/architecture-overview)). pg_sage cannot reimplement Neon's storage engine, but it can become the *autonomous operations layer* that the platforms below either lack or charge a premium for.

## 1. Zero-touch tuning of a freshly-spawned database

**The real problem at agent scale.** A fresh Postgres ships with upstream defaults that ignore the host: `shared_buffers` 128 MB, `work_mem` 4 MB, `max_connections` 100 ([PostgreSQL wiki](https://wiki.postgresql.org/wiki/Tuning_Your_PostgreSQL_Server)). An agent that spins up a 16 GB instance and walks away runs it as if it had a few hundred MB. The developer-platform Postgres offerings confirm this is the norm, not the exception: Railway and Fly.io run essentially stock Postgres and leave sizing to the user ([Encore: Render vs Railway](https://encore.dev/articles/render-vs-railway), [Software Scout: Fly vs Railway](https://thesoftwarescout.com/fly-io-vs-railway-2026-which-developer-platform-should-you-deploy-on/)). This is precisely the "often poorly tuned" case.

**Who solves it, how.** Managed services tune memory parameters to the *instance class* via formulas, not a human. Aurora PostgreSQL sets `shared_buffers` to 50–75% of instance RAM and sizes `autovacuum_work_mem` as `GREATEST(DBInstanceClassMemory/32768, 65536)` ([AWS re:Post](https://repost.aws/knowledge-center/rds-aurora-postgresql-shared-buffers), [Aurora tuning concepts](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraPostgreSQL.Tuning.concepts.html)). Cloud SQL auto-manages `shared_buffers` at ~33% of RAM but, tellingly, leaves `work_mem` at 4 MB and `random_page_cost` at 4.0 even on SSD — even managed defaults are only half-tuned ([Google Cloud SQL flags](https://cloud.google.com/sql/docs/postgres/flags), [oneuptime](https://oneuptime.com/blog/post/2026-02-17-how-to-optimize-cloud-sql-postgresql-performance-by-tuning-work-mem-and-shared-buffers-flags/view)). The PGTune heuristics are the public baseline: `shared_buffers` ≈ 25% RAM, `effective_cache_size` ≈ 50–75% RAM, `work_mem` derived from RAM/`max_connections` ([PostgreSQL wiki](https://wiki.postgresql.org/wiki/Tuning_Your_PostgreSQL_Server), [pgtune](https://github.com/impredicative/pgtune)).

**The pg_sage autonomous action.** At the moment a deployment transitions to `available`, run a **spawn-time tuning pass**: compute the PGTune-class parameter set from the instance's actual RAM/vCPU (already known from the `SizeProfile` and provider class), and apply it via the provider's native channel — an RDS/Cloud SQL parameter group for cloud instances, `ALTER SYSTEM`/`ALTER DATABASE` for local and Lakebase-branch DBs. This is the single highest-leverage zero-touch action because it requires no workload history — it is pure spawn-time arithmetic — and because the v1 fingerprint already tells pg_sage this is an agent DB, the tuner can bias defaults agent-appropriately (aggressive autovacuum, partitioned tool-call logs). The residual — `work_mem` and `random_page_cost` adapted from *observed* load — is the part managed services skip and is the natural second pass once the DB is in the fleet (see §7).

## 2. Autonomous indexing with verify-and-revert

**The real problem.** Agent-emitted schemas are stereotypically under-indexed (FK columns unindexed — Postgres doesn't auto-index them) and over-indexed (every "fact" gets an index, slowing writes), per v1. At fleet scale no human reviews any of it. The hard part is not *recommending* an index; it is applying one and **knowing whether to keep it** without a DBA watching.

**Who solves it, how — Azure SQL is the canonical precedent.** Azure SQL Database automatic tuning exposes `CREATE_INDEX`, `DROP_INDEX`, and `FORCE_LAST_GOOD_PLAN`. The loop: identify a candidate → apply **only during low CPU/IO windows** → **automatically verify a positive performance gain over 30 minutes to 72 hours** (longer for rarely-run queries) → **revert immediately on any regression** ([Microsoft Learn: Automatic Tuning](https://learn.microsoft.com/en-us/azure/azure-sql/database/automatic-tuning-overview?view=azuresql), built on the [Azure auto-indexing paper](https://www.microsoft.com/research/uploads/prod/2019/02/autoindexing_azuredb.pdf)). The guardrails are worth copying verbatim: won't create an index if it pushes the DB past 90% of max size; skips tables whose heap/clustered index exceeds 10 GB; `DROP_INDEX` only drops indexes unused for 90+ days and **never** drops unique/PK-supporting indexes; 21-day tuning history. Critically, **the verify-and-revert machinery only exists when the system applies the change** — apply the same DDL by hand and there is no validation ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/azure-sql/database/automatic-tuning-overview?view=azuresql)). Oracle Autonomous Database does the same with INVISIBLE/UNUSABLE candidate indexes validated in compile + execute phases, made visible only if they improve SQL and left invisible if they regress any statement ([Oracle: Autonomous Indexing](https://blogs.oracle.com/connect/post/autonomous-indexing)). **Aurora/RDS Postgres has no native auto-create-and-revert engine** — AWS offers only *detection* via DevOps Guru for RDS ([AWS docs](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/devops-guru-for-rds.html)) and customers bolt on pganalyze's advisor. That absence is the clearest competitive opening for a Postgres autonomous DBA.

**The pg_sage autonomous action.** pg_sage already has the executor primitives this needs: `CREATE INDEX CONCURRENTLY` on a dedicated connection, an auto-rollback goroutine with a rollback window, and regression detection ([REVERSE_SPEC §4](../docs/REVERSE_SPEC.md)). The work is to (a) wire those into the AgentDB path — today they only run against the primary fleet, not agent DBs — and (b) adopt Azure's guardrail set explicitly: act only in low-utilization windows, verify over a frequency-scaled window, never drop unique/PK indexes, age-gate drops at 90 days, cap by table size and free space. The reverse-spec notes pg_sage's regression detection currently uses *coarse global* metrics that mask per-table regressions; Azure's per-query verification is the standard to close that gap against. This is the feature competitors point customers at third parties for, and pg_sage already owns 80% of the machinery.

## 3. Auto-rightsizing for sporadic load

**The real problem.** A database idle 90% of the time then bursting is the worst case for static sizing: pay for peak 24/7, or under-provision and stall. Agent DBs are the archetype — sporadic usage is in the problem statement.

**Who solves it, how — two distinct strategies.** (a) *Serverless autoscaling* absorbs bursts in place. Neon autoscales 0.25–16 CU (1 CU = 1 vCPU + 4 GB), driven by an agent polling CPU/memory every 5 s and working-set every 20 s, targeting CPU ≤90% / memory ≤75% ([Neon autoscaling](https://neon.com/docs/guides/autoscaling-algorithm)). Aurora Serverless v2 scales 0–256 ACU (1 ACU ≈ 2 GiB) in 0.5-ACU steps and, since Nov 2024, scales to zero ([AWS scale-to-zero](https://aws.amazon.com/blogs/database/introducing-scaling-to-0-capacity-with-amazon-aurora-serverless-v2/)). (b) *Rightsizing advisors* for provisioned instances: AWS Compute Optimizer analyzes 32 days of CloudWatch CPU/IOPS/connections (and `DBLoad` with Performance Insights), needs ≥30 hours of metrics, and flags **idle** instances with stop/downsize/Graviton actions ([Compute Optimizer for RDS](https://aws.amazon.com/blogs/aws-cloud-financial-management/new-rightsizing-recommendations-for-amazon-rds-mysql-and-rds-postgresql-in-aws-compute-optimizer/), [32-day lookback](https://aws.amazon.com/about-aws/whats-new/2026/06/aws-compute-optimizer-ebs-ecs-32-day-lookback/)). The limitation of (b): advisors are reactive (need 30 h–32 d of history) and only *recommend*; for genuinely spiky load, scale-to-zero beats any static rightsizing.

**The pg_sage autonomous action.** pg_sage provisions on RDS/Cloud SQL — instance-class platforms, not serverless — so its rightsizing analog is the *advisor-that-acts*: collect CPU/IOPS/connection history per agent DB (once they're in the fleet), and when a deployment sits below a utilization floor for its lookback window, take the action Compute Optimizer only recommends — issue a provider resize down one class, or, for disposable/idle DBs, trigger the suspend/archive path (§4). The ROI math is already in [roi_and_market.md](./roi_and_market.md): one RDS class down is $132–$554/mo. The honest framing for the doc: where the underlying provider offers serverless (Aurora Serverless v2, Lakebase autoscaling branches), the *best* pg_sage action is to **recommend the workload move to a scale-to-zero tier** rather than rightsize a provisioned box — because pay-$0-when-idle beats any rightsizing for a DB idle 90% of the time.

## 4. Auto-archive of idle databases (TTL, cold storage, sprawl GC)

**The real problem.** Agents create databases per session at a rate where cleanup code "can fail silently, and in distributed systems, it eventually will" — one Neon setup showed **2,847 branches for a single database when it should have shown two** ([Neon: hidden ops layer](https://neon.com/blog/the-hidden-ops-layer-of-agent-platforms)). Two distinct moves get conflated and must be separated: *suspend compute* (cheap, reversible) versus *reclaim storage* (TTL delete or tier to object storage).

**Who solves it, how.** Neon separates them cleanly. Scale-to-zero suspends compute after 5 minutes idle and resumes in a few hundred ms — hibernation, not reclamation ([Neon scale-to-zero](https://neon.com/docs/introduction/scale-to-zero)). Reclamation is a *separate* branch-expiration/TTL feature: branches auto-delete after 1 hour / 1 day / 1 week, with "a periodic job to clean up expired branches" ([Neon: expire branches](https://neon.com/blog/expire-neon-branches-automatically)). The recommended belt-and-suspenders pattern is an **active GC sweep** ("a background job runs every 6 hours … queries for agent branches older than 24 hours") **plus TTL as a backstop** so a sweep failure stays bounded ([Neon: hidden ops layer](https://neon.com/blog/the-hidden-ops-layer-of-agent-platforms)). For genuinely cold data, the dominant tiering primitive is S3 Intelligent-Tiering — auto-demote at 30/90/180 days for up to 95% storage savings ([AWS S3 Intelligent-Tiering](https://docs.aws.amazon.com/AmazonS3/latest/userguide/intelligent-tiering-overview.html)). The limitation everywhere: "idle" is defined crudely as wall-clock since last query, which mis-reaps legitimately bursty workloads — the cure is **intent tagging** so the sweep distinguishes ephemeral from durable.

**The pg_sage autonomous action.** This is the closest to shipped: pg_sage already has leases (`lease_expires_at`), `ArchiveExpired`, TTL propagated to RDS tags and Lakebase branch TTL, and `ReconcileAbandonedDeployments` that archives then destroys lease-expired cloud instances behind a verified-backup gate ([reverse_spec §5.3, §6](../docs/reverse_spec/05-agentdb.md)). The two gaps to close: (1) **schedule the reconciler** — it is tested but nothing in `cmd/` calls it; a single goroutine on a 6-hour sweep (mirroring Neon's cadence) plus the existing advisory-lock guard turns dormant logic into an autonomous GC loop. (2) Make "archive" mean something — today it is a status, with explicitly no data movement. The autonomous action is a true **idle → suspend → export-to-S3 → destroy** ladder: detect idle from the agent ping gap (already recorded in `last_ping_at`), suspend/stop the provider resource, `pg_dump`-to-object-storage on the verified-backup path that already exists, then destroy — keeping the ledger row so a re-spawn can restore. Intent tags (`disposable` already exists on deployments) gate how aggressive the ladder is.

## 5. Multi-tenant noisy-neighbor isolation & auto-quarantine

**The real problem.** At agent scale, one tenant's burst — a large scan, a lock-holding transaction, connection exhaustion — starves every neighbor; this is "the most common cause of silent SLA breaches" ([Neon: noisy neighbor](https://neon.com/blog/noisy-neighbor-multitenant)). pg_sage's own multi-tenancy today is *tagging without isolation*: `tenant_id` is first-class but list endpoints return all tenants and there is no per-tenant resource governor ([reverse_spec §6](../docs/reverse_spec/05-agentdb.md)).

**Who solves it, how — and the load-bearing caveat: nobody solves it cleanly.** The isolation-model spectrum: Nile gives per-tenant *virtual* DBs inside one Postgres via session-scoped `set nile.tenant_id`, but openly states per-tenant resource quotas "are not yet implemented" ([Nile docs](https://www.thenile.dev/docs/tenant-virtualization/tenant-isolation)). Supabase gives a dedicated Postgres instance per *project* — best security blast radius, cost-prohibitive at thousands of tenants ([Supabase compute docs](https://supabase.com/docs/guides/platform/compute-and-disk)). Turso runs database-per-tenant as files on S3 at hundreds-of-thousands scale, but says plainly isolation "is good but it is not perfect," with open problems in "resource allocation and consumption" ([Turso architecture](https://turso.tech/blog/a-deep-look-into-our-new-massive-multitenant-architecture)). On the Postgres mechanism side, every control has a hole: `statement_timeout` per tenant caps runaway queries but not memory/IO; per-role connection limits prevent exhaustion but don't isolate CPU, and Postgres "does not immediately kill existing tenant connections when a new user limit is set"; Linux cgroup2 is the only structural CPU/mem/IO cap but the demonstrated approach throttles the *whole* Postgres service, not per-tenant backends ([Neon: noisy neighbor](https://neon.com/blog/noisy-neighbor-multitenant), [Percona: cgroup2](https://www.percona.com/blog/controlling-resource-consumption-on-a-postgresql-server-using-linux-cgroup2/)). The unavoidable conclusion: shared tenancy requires "constant firefighting" because Postgres lacks built-in per-tenant governors.

**The pg_sage autonomous action — auto-quarantine.** This caveat *is* the opportunity: since the platform layer can't isolate tenants, an observe-then-throttle control loop is where the value is. pg_sage already correlates by `application_name` (v1's per-agent identity foundation) and already has `pg_stat_activity`/`pg_stat_statements` collection. The loop: **detect** a single agent/tenant `application_name` consuming N% of locks/IOPS/CPU → **quarantine** by lowering *that role's* `statement_timeout` and `connection limit` (the two levers Postgres actually honors per-role), kill its in-flight runaway query, and (where pg_sage controls the host) reassign its backends to a constrained cgroup slice. This slots directly into the existing trust ramp: monitor (alert) → advisory (recommend) → auto (throttle). The honest design note for the doc: memory and IO caps remain *soft* on shared compute, so quarantine is statement-timeout + connection-cap + query-kill, not a hard guarantee — and that's still strictly more than any of Nile/Turso/Supabase offer autonomously today.

## 6. Usage metering & auto-chargeback

**The real problem.** pg_sage's chargeback is agent-self-reported via `AddCostSample` with **no automatic metering** — an agent can simply not report, and budgets are advisory ([reverse_spec §4.3](../docs/reverse_spec/05-agentdb.md)). At fleet scale, unattributed cost is the FinOps blocker.

**Who solves it, how.** Every serverless vendor invents a synthetic consumption unit so it can bill fractionally: Neon meters **CU-hours** (compute, only while active) + **GB-month** storage on copy-on-write deltas ([Neon pricing](https://neon.com/blog/new-usage-based-pricing)); Aurora meters **ACU-seconds** ([AWS docs](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/aurora-serverless-v2.how-it-works.html)); CockroachDB meters the **Request Unit** — "1 RU = 3 ms SQL CPU = 1 KiB egress," a SELECT ≈ 1–15 RU, a write ≈ 10–25 RU — which is the most directly per-statement-attributable model ([Cockroach Labs](https://www.cockroachlabs.com/docs/cockroachcloud/serverless-resource-usage)). The FinOps discipline on top: **showback** displays each tenant's charges, **chargeback** moves them to the tenant's P&L — an accounting choice, not a maturity ladder ([CloudZero](https://www.cloudzero.com/blog/chargeback-vs-showback/)). The field rule is to reach **≥80% of spend attributable** before flipping on hard chargeback ([FinOps Foundation: allocation](https://www.finops.org/framework/capabilities/allocation/)), and the emerging standard is **FOCUS** (FOCUS 1.3 ratified Dec 2025 added shared-cost allocation).

**The pg_sage autonomous action.** Replace self-reporting with **measured** metering: pg_sage already lives in the DB and collects `pg_stat_statements`, so it can reconstruct a per-tenant/per-agent consumption unit (CPU-ms + IO + storage-delta, keyed on `application_name`) without trusting the agent — the same data RU is built from. Roll it up nightly into a per-tenant, FOCUS-shaped showback record (the `agent_db_cost_samples` table is the natural home), and promote to enforced chargeback — flipping a deployment to `budget_exceeded` and throttling — once ≥80% of a tenant's compute is attributable. The key reframe versus today: cost becomes *observed by pg_sage* rather than *declared by the agent*, which is both more trustworthy and the prerequisite for the auto-quarantine loop in §5 (a tenant blowing its budget is the same signal as a noisy neighbor).

## 7. Config-template & drift enforcement across thousands of DBs

**The real problem.** Thousands of agent DBs drift from any intended baseline as divergent spawn paths and ad-hoc agent `ALTER`s accumulate — a reliability *and* cost problem, since drift means the §1 tuning silently rots.

**Who solves it, how.** Two layers. *Engine parameters:* RDS **parameter groups** are the desired-state primitive — "changes apply to all DB instances associated with that parameter group" — and AWS ships a pattern for automated change-monitoring that alerts on every parameter modification and which instances are affected ([AWS parameter groups](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_WorkingWithParamGroups.html), [AWS Database Blog](https://aws.amazon.com/blogs/database/automated-parameter-and-option-group-change-monitoring-in-amazon-rds-and-amazon-aurora/)). *Schema:* **Atlas** treats schema as declarative HCL and runs drift detection by "regularly comparing the actual database schema with the intended schema," emitting diffs and alerts; Atlas v1.0 (Dec 2025) added explicit **multi-tenant/fleet** support — define tenant groups and "plan and apply schema changes across many databases in a single operation" with canaries and parallelism ([Atlas drift detection](https://atlasgo.io/monitoring/drift-detection), [Atlas v1.0](https://atlasgo.io/blog/2025/12/23/atlas-v1)). The limitation everyone shares: Atlas drift detection *detects and reports* but does not auto-remediate, and RDS parameter changes often require a reboot to apply.

**The pg_sage autonomous action.** Define a **config template per size-profile** (the `agent_db_size_profiles` table already seeds defaults — extend it from sizing into full parameter baselines), then run a continuous **drift sweep** across the deployment fleet: compare each agent DB's live parameters against its profile's template and, at sufficient trust, *re-apply* the template — the auto-remediation Atlas leaves as a gap. Because pg_sage *acts* (not just alerts), it closes the loop that RDS parameter-group monitoring and Atlas both stop short of. Bulk application must be **canaried and rolling**, never all-at-once — apply to a representative cohort, verify health, then fan out — mirroring Atlas's rollout strategy, and reusing pg_sage's existing circuit breakers as the per-DB halt signal.

## 8. Per-tenant least-privilege & secure spawn defaults

**The real problem.** Fresh Postgres databases are *insecure by default*, and an agent that spawns one rarely hardens it. v1 covered PII/secret leakage *inside* agent memory; this is the complementary control-plane concern — the *grant model* of a freshly-spawned tenant DB.

**Who solves it, how.** The baseline hardening is well-established and mechanical: every role can create objects in `public` (Postgres's "biggest security footprint"), so `REVOKE CREATE ON SCHEMA public FROM PUBLIC` and `REVOKE ALL ON DATABASE ... FROM PUBLIC`; split roles by concern (`app_service` DML, `app_migrator` schema, `app_reader` read-only) and never run the app as a superuser, which has `BYPASSRLS` on by default that **cannot be changed** and silently defeats tenant RLS; set `ALTER DEFAULT PRIVILEGES` so future tables inherit the grant model ([Permit.io](https://www.permit.io/blog/implementing-fine-grained-postgres-permissions-for-multi-tenant-applications)). The tenancy-model choice (schema-per-tenant vs database-per-tenant vs RLS) determines the grant surface ([PlanetScale: tenancy in Postgres](https://planetscale.com/blog/approaches-to-tenancy-in-postgres)). The load-bearing caveat from §5: RLS is only as strong as the absence of a `BYPASSRLS` app connection, so for agent fleets database-per-tenant makes the isolation boundary the database itself rather than a policy that can be misconfigured.

**The pg_sage autonomous action.** Make secure defaults a **spawn-time enforcement step**, peer to the tuning pass in §1: on transition to `available`, auto-run the `REVOKE public` + split-role + `ALTER DEFAULT PRIVILEGES` hardening against the new DB, and refuse to hand the agent a `BYPASSRLS`/superuser connection. pg_sage already produces AWS Secrets Manager–managed master passwords and enforces SSL/encryption in plans; extending that into the role/grant layer is a natural continuation. This is bulk-applicable too — the §7 drift sweep can audit existing agent DBs for the open-`public` footprint and remediate it fleet-wide.

## Top 8 autonomous AgentDB-at-scale capabilities (ranked)

Ranked by leverage = (impact at scale) × (fit to existing pg_sage machinery) ÷ (build cost).

1. **Schedule the lifecycle reconciler (auto-archive idle DBs).** The logic is built, tested, and unscheduled. One goroutine on a ~6-hour sweep turns dormant code into an autonomous GC loop that bounds the "2,847 branches" sprawl problem. Lowest cost, immediate impact. (§4)
2. **Spawn-time zero-touch tuning.** Pure arithmetic on instance RAM at `available`; no workload history needed; v1's fingerprint biases it agent-appropriately. Fixes the "poorly tuned" case at birth. (§1)
3. **Measured per-tenant metering replacing self-report.** Reconstruct an RU-style unit from `pg_stat_statements` keyed on `application_name`. Prerequisite for both chargeback and quarantine; pg_sage already collects the inputs. (§6)
4. **Auto-quarantine noisy tenants.** Detect-then-throttle via per-role `statement_timeout` + connection cap + query-kill, on the existing trust ramp. The platforms admit they can't do this; pg_sage's vantage and executor can. (§5)
5. **Autonomous indexing with Azure-style verify-and-revert.** pg_sage owns 80% of the machinery (CONCURRENTLY, rollback goroutine); the work is wiring it to AgentDBs and adopting Azure's guardrails + per-query (not global) verification. The capability Aurora/RDS point customers to third parties for. (§2)
6. **Fleet config-template + drift auto-remediation.** Extend `agent_db_size_profiles` into parameter baselines; sweep, detect drift, re-apply via canaried rollout. The auto-remediation step Atlas and RDS both stop short of. (§7)
7. **Secure spawn defaults (least-privilege enforcement).** Mechanical `REVOKE public` + split-role hardening at `available`; bulk-auditable via the drift sweep. Closes the insecure-by-default gap agents never address. (§8)
8. **Auto-rightsize / recommend-serverless for sporadic load.** Advisor-that-acts for provisioned DBs (resize down on sustained low utilization); for serverless-capable providers, recommend moving the workload to a scale-to-zero tier — pay-$0-when-idle beats rightsizing for DBs idle 90% of the time. (§3)

## Open Questions

1. **Should pg_sage put agent DBs in the real fleet, or run a lighter "fleet-lite" pipeline for them?** Today they're a separate ledger. A full collector/analyzer per agent DB is the natural answer, but at thousands of ephemeral DBs the per-DB goroutine model may not scale — does AgentDB need a sampled/pooled collector distinct from the primary fleet's per-instance pipeline?

2. **One row per branch, or one row per logical DB with a branch pivot?** Unresolved from v1. With Lakebase autoscaling branches as pg_sage's cheapest provisioning path and Neon-style branch counts in the thousands, the dashboard model (1000 rows vs 1 row + pivot) and the metering model (per-branch CoW delta) both hinge on this.

3. **Is the right primitive a managed provisioner, or a layer on top of Neon/Turso?** pg_sage provisions instance-class RDS/Cloud SQL, but the agent-scale economics belong to scale-to-zero serverless. Should pg_sage *become* a thin control plane over Neon/Aurora-Serverless/Lakebase branches rather than provisioning provisioned boxes it then has to rightsize?

4. **How aggressive should the idle→destroy ladder be, and who can veto?** Wall-clock idle mis-reaps bursty workloads. Intent tags (`disposable` exists) help, but the kill-switch UX — agent HTTP endpoint? operator override? grace period? — determines whether teams trust auto-archival enough to leave it on `auto`.

5. **What's the metering unit of record, and does it need to be FOCUS-shaped on day one?** RU-style reconstruction is appealing but synthetic; CU/ACU-seconds map to real provider bills. Picking the wrong unit early ossifies chargeback. Does pg_sage emit FOCUS records now, or its own unit with a FOCUS adapter later?

6. **Can auto-quarantine ever be a hard guarantee, or is statement-timeout + connection-cap the honest ceiling on shared Postgres?** Memory/IO isolation needs cgroups, which need host control pg_sage may not have on managed providers. Is the quarantine story explicitly "soft on managed, hard only where we own the host," and is that acceptable to position?

7. **Does spawn-time tuning need a workload-adaptive second pass, and when?** Memory-to-instance sizing is solved at spawn; `work_mem`/`random_page_cost`/autovacuum-per-table need observed load. For an ephemeral DB that may live minutes, is the second pass ever worth running — or does the fingerprint let pg_sage ship a good-enough static agent-DB profile and skip adaptation entirely?

8. **Verification debt to clear before citing externally:** three numbers in the source material lean on third-party or unverified claims and should be re-confirmed before they appear in customer-facing material — Neon's "~300–500 ms" cold start (docs commit only to "a few hundred ms"), the "~1 s" branch-creation figure (docs say "instantly"), and the widely-repeated "97% of Neon branches are agent-created" (verifiable via the v1 SaaStr/Threads sources but *not* found in Neon's own current docs; the verifiable substitute is the "2,847 branches vs. two" incident).
