## Remaining Items

### 1. API Key ALTER SYSTEM Persistence (NEW — Needs Investigation)

The adversarial retest agent discovered that `sage.llm_api_key` set via `ALTER SYSTEM` does not persist after `pg_reload_conf()`. The value shows blank after reload, requiring per-session `SET`. This is likely caused by the `PGC_SUSET` context change (BUG-2 fix) — `PGC_SUSET` GUCs can only be set by superusers within their session, and `ALTER SYSTEM` writes to `postgresql.auto.conf` but the reload may not apply for SUSET context.

**Options:**
- A) Keep as-is (security feature: key never written to postgresql.auto.conf)
- B) Change context to `PGC_SIGHUP` but add a `show_hook` that masks the value
- C) Implement `sage.llm_api_key_file` GUC for file-based key loading

**Severity:** MEDIUM — Affects deployment workflow but not security or functionality.

### 2. BUG-3 Full Verification Pending

The "queries" snapshot category fix (NULL queryid filter + error logging) was applied but not fully verified with a running collector cycle against a database with pg_stat_statements data. The fix addresses the known root causes. Full verification requires loading test data and waiting for a collector cycle.

### 3. BUG-13 (Analyzer Segfault) — Not Addressed

The Phase 16 testing found the analyzer background worker segfaults (signal 11) every ~60 seconds after the first successful cycle. This bug was discovered AFTER the fix plan was written and is NOT addressed in v0.6.0-rc2. It requires separate investigation (likely memory corruption or use-after-free in analyzer C code).

**Severity:** CRITICAL — Must be fixed before any production deployment.

### 4. BUG-14 (analyzer_interval GUC Ignored) — Not Addressed

`sage.analyzer_interval` set to 3600 but worker always starts with 60s default. Not addressed in this fix cycle.
